# GitHub Org Admin Setup — LENS

> **Scope:** unblock the v0.2.0 release matrix at `github.com/mabo-du/lens`. The release jobs are currently stalled in `queued`/`in_progress` for 1.5–2 hr because the org-level third-party-action blocklist is set to **"Allow local actions only"** (the conservative default applied to new orgs since the 2022 GitHub security lockdown).
>
> **Audience:** a maintainer with **`admin:org`** scope on `github.com/mabo-du`. The actions are quick but cannot be performed by anyone with only repo-level `admin` — the org-level Actions / Allowlist settings live under the org's **Settings** page, not the repo's.
>
> **Read this before opening a maintainer issue:** if you haven't read SMOKE_TEST.md §0.1 and run `bash scripts/smoke-test.sh` against a fresh checkout locally, do that first — most stalled-matrix reports trace to a workflow YAML typo or a missing secret, NOT the org blocklist.

---

## 1. What you're seeing (diagnostic)

When the third-party-action blocklist is on, a release job stalls with no progress and the workflow run page shows:

* **Status:** `queued` → `in_progress` → still 0/4 matrix cells complete after 60+ min
* **Sub-step never advances past** `Setup Node.js` / `Install Rust` / `Build PDF plumber sidecar` / `Build Tauri app` (because the third-party action reference is what trips the policy check on first use)
* **No error toast** — the runner is just sitting in the queue, waiting on an admin override that won't come

The local-equivalent error (only seen if you try the failing action with `actions/checkout@<sha>` lifted): `Server Error: This workflow references actions that are not allowed by your organization's policy`.

---

## 2. The fix (4 clicks + 1 allowlist entry)

### 2.1 Navigate to org settings

```
https://github.com/organizations/mabo-du/settings/actions
```

As a left-side menu, expand **Settings** under the org name, then click **Actions** → **General**.

### 2.2 Open the **Allow specified actions** radio

Three radio buttons appear under **Policies > Actions permissions**:

1. ❌ **Allow all actions to run** (most permissive — anyone in the org can run any third-party action)
2. ❌ **Allow only GitHub-owned actions and a defined list of verified creators** (limbo: not what we want here)
3. ✅ **Allow specified actions and reusable workflows** (the conservative middle ground — explicit allowlist)

Select radio #3 — **Allow specified actions and reusable workflows** — and tick **Allow actions created by GitHub** (so `actions/checkout`, `actions/setup-node`, `actions/setup-python`, `actions/upload-artifact` continue to work without explicit entries).

### 2.3 Add an allowlist entry for each LENS third-party consumer

The release pipeline + CI provision these third-party actions explicitly by pinned commit SHA. The allowlist pattern matches by `owner/repo` + optional SHA pin:

| Owner/repo | Used at | Pattern to allow |
|---|---|---|
| `dtolnay/rust-toolchain` | release.yml `Install Rust` step | `dtolnay/rust-toolchain@*` |
| `Swatinem/rust-cache` | ci.yml `Rust cache` step, sqlcipher job | `Swatinem/rust-cache@*` |
| `tauri-apps/tauri-action` | release.yml `Build Tauri app` step | `tauri-apps/tauri-action@*` |
| `pypa/gh-action-pypi-publish` | release.yml `publish-pypi` job | `pypa/gh-action-pypi-publish@*` |

The `@*` SHA-wildcard suffix is intentional — once an action is allowlisted, GitHub allows all currently-pinned SHAs inside that action's ref namespace, including new SHAs the workflow author pins to in subsequent releases. If you want strict per-SHA lockdown, write each pinned SHA explicitly (see §3 for the maintenance trade-off).

Add 4 entries (one per row above) to the **Allow specified actions** text input.

Click the green **Save** button at the bottom of the form.

### 2.4 Verify the lift (within 30 s)

Re-trigger a stalled release workflow run: open the failed run in **Actions**, click **Re-run all jobs**, then watch the matrix. If 4/4 cells complete within 15 min, the lift worked. If they're still stalling, jump to §4.

## 3. Maintenance: keeping the allowlist honest

When the workflow YAML bumps a pinned action SHA (the `Round-N followup` comments in `release.yml` and `ci.yml` document every such bump), the GitHub SHA-validating for ***new SHAs*** still works as long as the allowlist pattern uses `owner/repo@*`. If you instead choseto enumerate `owner/repo@<sha>` for each pin, you'll need to re-touch this admin page on every SHA bump — high-friction, easy to forget, easy to silently re-stall the matrix.

**Recommended:** keep the allowlist as `owner/repo@*` patterns (the explicit `@*` is enough to express the wildcard intent; the org-wide policy on third-party creators is enforced separately via the checkbox and the `Allow GitHub-created actions` toggle).

When **adding** a new third-party action (e.g. a future `cargo-binstall`-style helper), author the workflow YAML *first*, run the workflow to confirm the SHA-pin is valid, then come back to this page and `owner/repo@*` the new entry.

When **retiring** a third-party action (deleting the workflow step), retire its allowlist entry at the same time. Stale allowlist entries are inert (the workflow no longer references them) but they broaden the org's attack surface for future PR authors. Audit quarterly (see §4).

---

## 4. Quarterly audit

For each LENS release window, an org-admin maintainer should:

1. **Inventory:** `cat .github/workflows/*.yml | grep -oE 'uses: [^/]+/[^@]+'` — every distinct `owner/repo` pair across all 4 workflow files.
2. **Compare with the allowlist:** the entries on this page should be a strict superset of the inventory (each `owner/repo` show up EITHER through the **Allow GitHub-created actions** toggle OR as a `owner/repo@*` allowlist entry).
3. **Re-run release-dry-run:** trigger `.github/workflows/release-dry-run.yml` from the Actions tab with `platform=auto`. If it exits green, the matrix compatibility is verified. If it fails on a specific cell, the failure is a YAML/SHA-pinning issue, NOT an admin-policy issue, and you should look at the cell-specific log.

---

## 5. What to do when §2.2 the radio button is greyed out

If the radio buttons under **Allow specified actions and reusable workflows** are greyed out / disabled, you're probably either:

* not signed in with an **`admin:org` scope** account (repo-level `admin` is not enough — confirm under `https://github.com/mabo-du/people` and check the badge column);
* in an **enterprise-managed org** where the org policies are locked by the enterprise owner; you need to escalate to the enterprise admin to lift (mabo-du's enterprise owner has to do the lift centrally for all child orgs). If this is your situation, route the request through #enterprise-admins on Slack with link to this runbook.
