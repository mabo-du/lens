# **Architectural Evaluation of Rich Text Frameworks for Qualitative Data Analysis**

## **1\. Introduction: The Standoff Markup vs. DOM Paradigm Clash**

The engineering of a free, open-source desktop qualitative data analysis (QDA) application within an Electron and React environment presents one of the most formidable software architecture challenges in modern frontend development. At the absolute core of any QDA methodology—whether grounded theory, thematic analysis, or discourse analysis—is the necessity to assign highly granular, non-hierarchical, and densely overlapping semantic codes to unstructured text. Unlike conventional word processors or content management systems, where typographical formatting (such as bold or italic text) is applied sequentially and almost exclusively in a nested hierarchy, a QDA tool must seamlessly process multiple, concurrent, and conflicting annotations on identical or intersecting text spans. In a live coding session, a single sentence may carry five distinct thematic codes assigned simultaneously by three different researchers, all with differing start and end character offsets.  
This fundamental requirement introduces severe friction when deployed atop standard web rendering technologies. The browser's Document Object Model (DOM) is inherently hierarchical and strictly tree-structured. It natively prohibits overlapping, non-nested elements. A theoretical HTML structure such as \<mark\_alpha\>text \<mark\_beta\>intersect\</mark\_alpha\> text\</mark\_beta\> is fundamentally invalid markup; the browser parsing engine will forcefully restructure it, destroying the intended semantic boundaries. In the context of qualitative analysis, annotations must be stored immutably as mathematical character offsets against a canonical plain-text snapshot. Therefore, the visual representation within the rich text editor must perfectly and deterministically round-trip back to strict mathematical coordinates without data corruption, DOM pollution, or offset drift.  
Compounding this architectural complexity is the project's requirement to handle highly expansive documents—frequently exceeding 30,000 to 50,000 words—without experiencing performance degradation during live interaction. Furthermore, the editor must maintain absolute structural stability when rendering bidirectional (Bidi) scripts, such as Arabic, Hebrew, and Urdu, where the logical memory order of characters diverges radically from their visual presentation order.  
This comprehensive technical report provides an exhaustive evaluation of three primary rich text editor frameworks—ProseMirror, Slate.js, and Meta’s Lexical—to determine their precise architectural suitability for this highly specific use case. The analysis systematically deconstructs their internal data models, character offset preservation mechanisms, and rendering pipelines. The report extends to an examination of existing QDA-adjacent annotation systems (such as Hypothesis, Recogito, and INCEpTION) to identify proven, production-ready architectural patterns, culminating in a definitive architectural recommendation for the bespoke rendering strategies required to achieve the project’s objectives.

## **2\. Overlapping Annotation Rendering Architecture**

The most profound technical hurdle in rendering multiple QDA codes is resolving the impedance mismatch between standoff markup (where annotations are defined purely by numerical start and end offsets independently of the text content) and the hierarchical DOM tree. When two semantic spans overlap, rich text editors must artificially split the underlying text node into multiple contiguous leaf nodes to represent the intersection, resulting in unavoidable DOM fragmentation.

### **2.1 ProseMirror: Decoration Sets and Text Chunking**

ProseMirror addresses the challenge of inline styling through two distinct mechanisms: Marks, which are serialized and become a permanent part of the document's schema, and Decorations, which are purely transient, visual, and non-persistent overlays. For a heavily annotated QDA application, Decorations (specifically Decoration.inline) represent the only viable approach, as they do not permanently mutate the underlying text model, thereby preventing the persistent document schema from bloating with user-specific analysis metadata.1  
When overlapping inline decorations are applied programmatically, ProseMirror’s highly optimized rendering engine automatically divides the text content into discrete chunks based precisely on the decoration boundaries.1 Each resulting text chunk is subsequently wrapped in a separate DOM node containing the respective combination of decoration attributes. However, ProseMirror does not natively synthesize or merge the visual properties of these intersecting chunks. If a researcher highlights a word in yellow (Code A) and partially overlaps it with a blue highlight (Code B), ProseMirror will physically create a DOM node for the intersecting segment, but the developer is entirely responsible for supplying the custom computational logic to style the overlapping segment (e.g., detecting the collision and computing a blended color or stacked border UI).3  
Attempting to bypass this by mutating the DOM directly or forcibly modifying the decoration objects in memory will trigger immediate rendering failures, as ProseMirror strictly controls its state representation.4 A critical known limitation within ProseMirror's decoration architecture involves cursor placement and selection boundaries. When a user's selection is completely inside a mark with an inline decoration, and the selection focus leaves either the left or right side of the mark, the cursor can exhibit unexpected jumping behaviors, or the decorations can briefly detach. This instability is uniquely pronounced when manipulating the selection via mouse drag events rather than precise keyboard navigation, requiring complex plugin interceptors to stabilize the $anchor and $head positions during active dragging.5

### **2.2 Slate.js: Leaf Rendering and the Attribute Collision Flaw**

Slate.js groups contiguous text into "leaves" during the React rendering phase. When text-level formatting is rendered, characters sharing the exact same set of formatting attributes are grouped into a single leaf node.6 When overlapping decorations—which Slate manages dynamically at render time—intersect and force a single text node to split, the custom renderLeaf function provided by the developer receives a specialized leafPosition property.6 This object contains the specific start and end offsets of the newly fragmented leaf relative to the original, undivided text node, conceptually providing the developer with the necessary context to reconstruct the intersection.6  
Despite this theoretical capability, Slate.js harbors a severe architectural limitation regarding identical semantic marks, rendering it exceptionally hostile to QDA workflows. In version 0.47, the Slate core team made the controversial decision to remove the dedicated Mark interface entirely.8 Consequently, all custom decorators and marks are now merged directly into the leaf object using JavaScript's native Object.assign method.10  
Because Object.assign inherently overwrites duplicate object keys, it is fundamentally impossible for two marks of the exact same type but differing metadata to coexist on a single leaf. If a text span contains a user's highlight mark identified by { highlight: 'Code\_Alpha' } and an overlapping highlight mark identified by { highlight: 'Code\_Beta' }, the subsequent assignment will silently overwrite the previous one, destroying the overlapping researcher's code entirely.10 To circumvent this architectural blockade, developers are forced to utilize highly unoptimized arrays (e.g., { highlights: }), which entirely bypasses Slate's native intersection logic, requiring the engineering team to manually write computationally expensive, intersection-aware leaf creation algorithms inside the highly sensitive renderLeaf React render loop.10 This issue (tracked under GitHub issue \#3383 https://github.com/ianstormtaylor/slate/issues/3383) effectively disqualifies Slate's out-of-the-box decoration engine for multi-researcher QDA environments.10

### **2.3 Lexical: MarkNodes and the Native Nesting Restriction**

Meta’s Lexical framework approaches annotations through its @lexical/mark package, utilizing the abstract MarkNode.11 The MarkNode functions by attempting to distribute an inline wrapper across a heterogeneous run of child nodes, utilizing internal methods like $distributeInlineWrapper to lift any block-level children out of the hierarchy while maintaining the wrapper tightly around the leaf inline content.12  
While Lexical is structurally capable of splitting underlying text nodes via $splitNode() to accommodate intersecting boundaries 11, the framework's core philosophy strictly rejects nested elements unless explicitly circumvented through unmanaged DecoratorNode implementations.11 When Lexical forces overlapping MarkNodes to generate sibling text nodes rather than nested hierarchies, the internal Abstract Syntax Tree (AST) becomes highly volatile. As documented by the Lexical community (GitHub issue \#1839 https://github.com/facebook/lexical/discussions/1839), attempting to programmatically modify a TextNode while two distinct MarkNodes overlap it causes the path references to mutate unexpectedly.13 The secondary mark loses its original anchorNode/focusNode coordinate reference, silently breaking the annotation state and detaching the highlight from the DOM.13 This fragility during active AST mutation makes Lexical exceptionally risky for a QDA tool where users are constantly deleting, merging, and overlapping dense annotation clusters.

| :---- | :---- | :---- | :---- |
| **ProseMirror** | Dynamic DOM splitting via Decoration.inline. | Developer must compute combined classes. | High DOM fragmentation; documented mouse selection bugs during overlap traversal.3 |
| **Slate.js** | Leaf generation via renderLeaf and leafPosition. | Custom arrays needed inside leaf attributes. | Object.assign mutation overwrites identical semantic marks; Issue \#3383.10 |

## **3\. Character Offset Preservation and Canonical Fidelity**

In the domain of qualitative data analysis, the rich text visual editor is merely a temporary, interactive projection. The ultimate, authoritative source of truth is the canonical plain-text document alongside the mathematical coordinate pairs (start\_char, end\_char) representing the researcher's annotations. An editor framework must possess the architectural integrity to ingest raw text and standoff offsets, map them deterministically to its internal AST, permit active editing, and flawlessly translate the internal model back to absolute plain-text character offsets upon saving, without losing a single integer of fidelity.

### **3.1 ProseMirror: Token Indices and Linear Mapping**

ProseMirror operates on a highly unique token index system rather than a strict, DOM-like path hierarchy. It elegantly represents positions within the document using simple integers, treating the entire hierarchical document as a flattened sequence of tokens.14 In this architecture, a token can represent a visible text character, the opening tag of a structural node, or the closing tag of a structural node. Position 0 always points to the absolute start of the document, and every subsequent position increments this linear index.14  
However, ProseMirror's document positions do not map one-to-one with standard string character offsets. As articulated by the framework's creator, the concept of "raw text coordinates doesn't really mean anything" internally, because the number of ProseMirror structural tokens residing between two pieces of visible text frequently does not correspond to the number of raw newlines (\\n) or invisible markup tags in a raw text string.16 To round-trip absolute character offsets successfully, a QDA application must engineer a dedicated translation layer. The mathematically sound approach is to map the text as a continuous set of (offset, string) pairs—tracking one per textblock—and then dynamically add the block's start offset to the localized string offset when transitioning coordinates back into the ProseMirror schema.17 ProseMirror excels here through its implementation of the StepMap class. Transactions cleanly map through these step maps, enabling an external, piece-table-based offset tracker to stay perfectly synchronized as the user edits the document, effortlessly handling structural rebasing.2

### **3.2 Slate.js: Path Tuples, Offsets, and AST Fragility**

Slate.js diverges significantly by tracking internal positions utilizing a combination of a Path (an array of integers representing the exact traversal indices down the node tree) and a highly localized offset representing the character position within that specific text leaf.19 The Transforms API enables moving cursors, expanding selections, or inserting nodes by specifying standard text units like offset or character.20  
While this model closely mirrors the mental model of the traditional DOM, it proves to be exceptionally fragile during heavy document mutations. Because Slate relies fundamentally on coordinate paths, any user operation that inserts a character, deletes a word, or splits a node immediately shifts the paths of all subsequent nodes in the document. Slate's internal history plugin attempts to track and map these operations automatically, but complex overlapping highlight calculations or external standoff modifications frequently result in operations pointing to stale or entirely invalidated paths.23 When this occurs, Slate will crash the entire React application with the error "Cannot resolve a DOM node from Slate node" (GitHub Issue \#5281).23 Converting absolute plain-text offsets to Slate paths requires an aggressive, recursive walk of the entire Slate tree, summing text lengths until the desired offset is mathematically reached. In a 30,000-word interview transcript, performing this recursive AST walk continuously during active typing induces massive computational overhead.

### **3.3 Lexical: The Deprecation of Offset Abstractions**

Lexical previously provided a dedicated package, @lexical/offset (specifically the OffsetView class), intended precisely to handle the translation of internal selection offsets to and from absolute character indices.25 However, in a major architectural shift, the Lexical core team explicitly deprecated and removed this package in version 0.44.0. The release notes stated unequivocally that OffsetView "never worked correctly, has no documentation, no tests, and no usage in the monorepo".25  
The total deprecation of @lexical/offset constitutes a catastrophic liability for any QDA tool attempting to use Lexical. It violently forces the engineering team to build a completely bespoke, highly complex offset resolution engine directly on top of Lexical's constantly reconciling internal AST. While Lexical retains the basic ability to split nodes by character offsets via $splitNode() 11, accurately calculating the absolute offset of an annotation across a heavily decorated, multi-node Lexical document requires complex traversals (utilizing legacy abstractions like $createChildrenArray) that are no longer officially maintained, documented, or supported by Meta.25 Attempting to maintain offset fidelity without framework-level support introduces unacceptable risks of data corruption when persisting qualitative codes to the database.

## **4\. Performance Profiles for Heavy Annotation Workloads**

A standard 30,000-word qualitative interview transcript, when subjected to rigorous analysis by multiple researchers, will easily accumulate upwards of 300 distinct, non-hierarchical, and overlapping codes. Because all three rich text frameworks fundamentally resolve overlapping annotations by splitting the underlying text nodes into smaller, contiguous DOM fragments, a single densely coded paragraph could theoretically shatter into dozens or hundreds of disparate HTML \<span\> elements.

### **4.1 DOM Fragmentation, Layout Thrashing, and Rendering Bottlenecks**

The primary performance bottleneck in web-based rich text editors is not the JavaScript execution speed, but the sheer volume of generated DOM nodes. When the browser's rendering engine is forced to calculate layout constraints, repaint, and reflow thousands of fragmented leaf nodes to support overlapping highlight visuals, the main thread becomes severely blocked.  
ProseMirror applications have historically faced profound rendering latency when loading massive documents (e.g., rendering the full text of *Moby Dick* within a single contenteditable container), particularly in browsers utilizing less optimized painting engines, such as Firefox running on macOS or Linux.29 However, ProseMirror’s pure separation of state and view allows for aggressive React memoization wrappers, which significantly reduce the React reconciliation overhead.29  
To further mitigate the catastrophic DOM node explosion in large documents, advanced implementations rely heavily on modern CSS layout boundaries. By applying content-visibility: auto and contain-intrinsic-size to the block-level wrapper nodes (such as paragraphs), the browser is instructed to entirely defer the layout and painting computation of off-screen text blocks. This CSS architecture dramatically increases the framerate in Chromium-based environments like Electron (despite historical Chromium bugs like \#1141742, which have since stabilized).30 Additionally, developers have found that disabling the native browser spellchecking engine (spellcheck: 'false') yields massive performance improvements, as the browser is no longer forced to run dictionary lookups across thousands of artificially fragmented \<span\> boundaries.30

### **4.2 The Absence of True Virtual Rendering**

In an extreme stress test configuration—where a document contains a distinct semantic mark on every single word—ProseMirror's internal view layer will experience severe degradation if the user does not naturally divide their text into paragraph blocks.31 Currently, ProseMirror assumes that every mark in the AST has a corresponding, physically rendered DOM node.31 True virtual rendering (the ability to calculate marks purely in memory without outputting them to the active DOM) is not natively supported without completely overriding the internal serializeNodeAndMarks pipeline, an endeavor that borders on rebuilding the editor from scratch.31  
Slate.js offers a functional escape hatch via the shouldNodeComponentUpdate function, which allows developers to manually intercept the React rendering lifecycle and force the framework to skip updates for nodes that have not mathematically changed.8 While this optimization dramatically improves responsiveness during active typing, the initial bootstrap and parsing of a heavily split document in Slate is notorious for causing main thread blocking, as the initial React tree generation cannot be bypassed.33  
Lexical's highly touted reconciliation algorithm is heavily optimized for batched DOM updates 11, but the core physical limitation remains insurmountable: the browser's rendering engine must eventually draw every single fragment. Because Lexical restricts nested elements 12, complex overlapping annotation graphs force extreme horizontal fragmentation, degrading layout calculation speeds at an exponential rate as the document grows.

## **5\. Architectural Analysis of QDA-Adjacent Annotation Systems**

To architect a robust QDA editor, it is vital to analyze how existing, production-hardened open-source web annotation systems successfully resolve the overlapping markup problem. A definitive pattern emerges: these tools frequently bypass standard rich-text DOM constraints entirely by mathematically separating the canonical text layer from the visual annotation layer.

### **5.1 The Hypothesis Client Architecture**

Hypothesis (https://web.hypothes.is/) operates as a dynamic overlay client rather than a traditional WYSIWYG text editor.34 The backend service is built in Python utilizing the Pyramid framework, while the client side utilizes specialized, standalone DOM anchoring libraries (dom-anchor-text-quote, dom-anchor-text-position) to mathematically calculate where annotations belong on a static host page, including complex PDF rendering via Mozilla's PDF.js.34  
While this architecture is highly effective for passive reading environments, Hypothesis suffers from severe race conditions when deployed in interactive, heavily mutated documents. A known, fundamental architectural bug in the Hypothesis client (documented in GitHub issue https://github.com/hypothesis/h/issues/3278) occurs when overlapping annotations are highlighted concurrently; the DOM can mutate in the brief microsecond window between an annotation being mathematically located and its corresponding visual ranges being drawn.39 This race condition corrupts the subsequent highlights, reinforcing the extreme danger of relying purely on dynamic DOM mutation to render overlapping highlights in an editable environment.39

### **5.2 RecogitoJS and text-annotator-js**

RecogitoJS (historically located at https://github.com/recogito/recogito-js) and its official, modernized successor, the Recogito Text Annotator (https://github.com/recogito/text-annotator-js), were engineered specifically for Linked Data web annotation.40 text-annotator-js handles overlapping selections natively through a robust configuration property (selectionMode: 'shortest' | 'all'), which mathematically defines the browser's behavior when a user clicks overlapping span coordinates.44  
However, text-annotator-js is explicitly an annotation overlay, not a rich text *editor*. It is designed to be applied over static, read-only text.40 Because the library forcibly manipulates the DOM to inject physical HTML spans (such as \<mark\> tags) to render highlights 45, it suffers from the exact same layout thrashing issues as rich text editors when text edits occur. The maintainers themselves note that mutating the DOM directly to maintain coordinate fidelity becomes "a huge mess" when dealing with overlapping and deeply nested highlights.46

### **5.3 INCEpTION and the Brat Visualization Paradigm**

INCEpTION, a highly advanced, institutional-grade semantic annotation platform, tackles the overlap problem using a wholly different architectural paradigm. Rather than forcing the brittle HTML DOM to represent overlapping text annotations via fragmented \<span\> elements, INCEpTION utilizes the brat rapid annotation tool (originally brat.nlplab.org) as one of its primary frontend rendering engines.47  
brat renders the document text strictly visually using Scalable Vector Graphics (SVG).48 The text content is laid out in a read-only container, and annotations (including highly complex overlapping spans, hierarchical relations, and directional arcs) are drawn mathematically as dynamically positioned SVG shapes overlaying the text.48 This pure "standoff markup" approach completely decouples the visual annotation presentation from the text's physical DOM hierarchy, flawlessly bypassing DOM fragmentation limitations.51 INCEpTION optimizes extreme payloads by pre-rendering complex annotated documents on its Java server backend into a condensed JSON visual representation (containing precise span offsets, arc endpoints, and colors) before transmitting it to the JavaScript client via RESTful HTTP protocols, making client-side rendering nearly instantaneous regardless of document size.50

## **6\. Bidirectional (Bidi) Text Stability Comparison (2025-2026)**

Qualitative research frequently involves highly diverse global data. Supporting mixed Right-to-Left (RTL) and Left-to-Right (LTR) text streams introduces the Unicode Bidirectional Algorithm (UBA), an incredibly complex set of resolution rules (such as rules P2, P3, X9, and N0-N2) that the browser's layout engine uses to dynamically determine text flow.53 Standard web editors are overwhelmingly "LTR-Brained," treating RTL as a superficial, cosmetic CSS property (direction: rtl) rather than a foundational architectural paradigm.54

### **6.1 The Cursor Jump and Punctuation Drift Dilemmas**

When an editor is architecturally unaware of the UBA's distinction between logical memory ordering and visual presentation ordering, crossing a script boundary (for example, typing an English technical term inside an Arabic sentence) causes the cursor to jump erratically to the wrong logical position.54 The editor’s internal state fundamentally disagrees with the browser’s visual rendering pipeline.  
Furthermore, because QDA tools fragment the DOM to render overlapping highlight spans, they violently interrupt the UBA's directional context resolution. Punctuation characters (such as spaces, commas, and periods) are classified as "neutral" in the Unicode specification, meaning they dynamically derive their visual direction from the surrounding strong characters.54 If a rich text editor splits a text span immediately before a period to apply a colored highlight, the period is isolated from its preceding Arabic text, loses its RTL context, and will visually drift to the opposite side of the sentence, rendering the text completely unreadable for native speakers.54

### **6.2 CSS Isolation and Critical Electron WebView Bugs**

To prevent fragmented DOM spans from ruining the overall Bidi layout, developers often attempt to apply unicode-bidi: isolate or unicode-bidi: plaintext to the injected spans.55 The unicode-bidi: isolate property forces the browser to compute the span's internal directionality independently of its parent, preventing neutral characters from bleeding their directionality.55 However, heavily isolating spans in cursive scripts like Arabic entirely destroys critical typographic ligatures, as the browser treats the isolated span boundary as a hard, interrupted visual block, severing connected letterforms.  
Crucially, desktop applications built in Electron (or Tauri) rely on embedded WebViews, which possess a documented, severe rendering bug regarding RTL span splitting. If an RTL paragraph containing heavily fragmented inline spans naturally wraps to a new line, the WebView layout recalculation triggers an uncontrollable, instantaneous auto-scroll behavior. Even aggressively intercepting scroll events via e.preventDefault() or stopImmediatePropagation() entirely fails, because the scroll is executed deep within the WebView's C++ layout engine during reflow, bypassing the JavaScript V8 event loop entirely (documented extensively in Tauri Issue \#15330).58

## **7\. Lexical Deep Dive: The Emerging Third Candidate**

Meta’s Lexical framework was designed specifically to resolve the accessibility and performance bottlenecks of its predecessor, Draft.js. For a new QDA project, its batched update architecture and strict functional paradigms initially appear highly attractive. However, evaluating it against the strict demands of overlapping QDA codes reveals systemic, unresolvable shortcomings.  
Lexical’s architecture for inline decorations revolves around @lexical/mark. It natively accommodates standard textual formatting and basic inline annotations via the MarkNode.11 However, Lexical fundamentally rejects nested elements unless explicitly circumvented through the usage of unmanaged DecoratorNode mechanisms.11 When forced to handle overlapping marks, it must aggressively generate sibling text nodes. As demonstrated in community bug reports, attempting to modify an underlying TextNode while two marks overlap it permanently destroys the coordinate references for the second mark.13  
Most critically, the absolute deprecation of the @lexical/offset library in version 0.44.0 indicates unequivocally that the Lexical core maintainers do not view strict, round-trippable character offsets as a first-class framework feature.25 Without reliable offset mappings, rebuilding the standoff annotations required by a QDA tool becomes an immense, unsupported engineering burden. Furthermore, despite its growth since 2022, the broader community plugin ecosystem for QDA-specific, heavily annotated workflows does not yet exist for Lexical; it remains heavily tailored towards social media text entry, basic CMS word processing, and chat interfaces.63

## **8\. Final Architectural Recommendation**

Selecting a rich text editor framework for a complex Qualitative Data Analysis application involves choosing the architectural foundation with the fewest fatal compromises.

* **Slate.js** is fundamentally disqualified by its Object.assign architectural limitation on identical semantic marks 10, rendering it functionally incapable of handling multiple overlapping researchers' codes cleanly. Its path-based mapping is also highly fragile under heavy programmatic mutation, risking application crashes.23  
* **Lexical** is disqualified due to the explicit deprecation of its offset calculation libraries 25, severe AST instability when modifying overlapping MarkNodes 13, and its strict anti-nesting paradigms which limit markup flexibility.11

**ProseMirror** emerges as the definitive, undisputed recommendation for this Electron/React project. While it requires a steep learning curve and substantial custom implementation, its core architecture—utilizing integer-based token indexing 14, immutable state transactions, and robust React rendering wrapper support 29—provides the exact mathematical stability required to map active DOM states back to plain-text standoff offsets without data corruption.

### **Recommended Implementation Blueprint**

To successfully deploy ProseMirror for a QDA tool supporting non-hierarchical overlapping codes, 50,000-word documents, and complex Bidi text within an Electron/React environment, the out-of-the-box API will not suffice. The following custom architectural pattern must be implemented:

1. **The Standoff Mapper (Piece Table Architecture):** Do not attempt to serialize the active ProseMirror DOM back to HTML or Markdown for storage, as formatting collisions will cause data loss. Instead, maintain a secondary, invisible "Piece Table" or offset map. Utilize ProseMirror’s native StepMap interface 2 to actively listen for transaction steps, calculating the exact delta in token integers and mathematically applying them to the character offsets stored in the local database.17  
2. **Decoupled Rendering via SVG Underlay (The INCEpTION/Brat Pattern):** To utterly prevent DOM fragmentation, do not use ProseMirror's Decoration.inline to apply physical background colors to text spans. This standard approach will cause thousands of \<span\> elements to render, destroying Bidi typographic ligatures 54 and triggering the catastrophic Electron RTL scroll bug.58 Instead, inject the text cleanly into ProseMirror as uniform text blocks. Utilize a React overlay component to query the browser's Range.getBoundingClientRect() API for the absolute pixel coordinates of the annotation character offsets. Draw the overlapping highlights mathematically on an absolute-positioned, pointer-event-disabled SVG or Canvas layer located precisely behind the transparent ProseMirror text layer. This successfully replicates the highly performant INCEpTION/brat architecture.48  
3. **Aggressive Performance Optimization:** Implement modern content-visibility: auto and contain-intrinsic-size CSS properties on all block-level nodes via customized ProseMirror nodeViews to defer off-screen layout calculations.30 Forcefully disable native browser spellchecking (spellcheck: 'false') during heavy live-coding sessions.30  
4. **Bidi Support Wrappers:** Utilize the proven logical structures from @remirror/extension-bidi to forcefully manage the dir attribute strictly at the paragraph/block level.59 By relying on the decoupled SVG underlay for visual highlights instead of physical \<span\> injection, the browser's native Unicode Bidirectional Algorithm will function entirely without interruption, completely eliminating cursor jumps, punctuation drift, and rendering anomalies.
