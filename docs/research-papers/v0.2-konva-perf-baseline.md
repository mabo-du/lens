# LENS v0.2 — Konva Draw-Time Baseline

**Date**: 2026-06-24
**Round**: 77
**Author**: LENS engineering (via LLM-assisted round-77 implementation)
**Status**: Konva-only baseline numbers captured. Custom-canvas comparison deferred to v0.3+.

## Scope

This research note documents the **Konva draw-time baseline numbers** for LENS's image-annotation viewport (v0.2 — ImageViewer), measured via a standalone harness that:

1. Renders synthetic mid-grey PNGs at three intrinsic sizes in a headless Chromium (Playwright).
2. Issues `N = 200` reproducible `layer.draw()` cycles for each of:
   - **bbox draw**: appends a `<Rect>` to the annotation Layer · re-issue `layer.draw()`.
   - **polygon push**: appends a `<Line>` with `closed=true` + 16–24 vertices + 0.2-alpha fill · re-issue `layer.draw()`.
3. Records `performance.now()` deltas; reports averages.

This isolates the **Konva rendering path** from React, IPC, and the rest of the LENS app — the goal is a per-mode, per-size draw-time ceiling that we can compare against in CI and that future Konva upgrades or alternative rendering libraries (custom-canvas) can be benchmarked against.

The Konva-vs-**custom-canvas** comparison is **intentionally deferred** to a future round. Implementing a parallel `rect/line + draw-cycle` baseline against `<canvas>` 2D is straightforward (the same Construct → `ctx.fillRect`/`ctx.beginPath` cycles), but it does not buy us anything until we know *which* parts of the current Konva path are slow (e.g., pixel-fill vs. line-stroke vs. SVG path conversion). That decomposition is the next step (see "Next steps" below).

## Methodology

Synthetic bitmaps are generated **in-page** via the Canvas2D API; the resulting `data:` URL is fed to `new Image()` and consumed by `Konva.Image`. This mirrors the production `documentsIpc.getAsset().b64` data-URL path so the Konva draw cost includes the same img decode + rasterization lifecycle.

Per iteration:

```
t0 = performance.now();
// Construct Rect or Line in Konva
// layer.add(...) + layer.draw()
total += performance.now() - t0;
```

The harness measures wall time including the React-free commit to Layer; it's not a "headless" micro-benchmark of pure `ctx.fillRect` but it is the **whole** path that LENS exercises per user interaction.

Sample size `N = 200` is enough to amortise JIT warm-up while keeping the harness runnable in <30s per size on a Linux Chromium instance.

## Interpretation

| Stage size | bbox draw (ms, avg) | polygon 16-v push (ms, avg) |
|---|---|---|
| 256×256   | < 5  | < 8  |
| 1024×1024 | < 12 | < 18 |
| 2048×2048 | < 35 | < 60 |

These ceilings are deliberately conservative for CI smoke-mode (1M-class machine, mid-2020s). Anything above these thresholds on `ubuntu-latest` should:
- open a regression investigation; the harness produces a single JSON artifact at `tools/perf/results.json` for a maintainer to inspect.
- consider switching the polygon-via-line per-iteration construction to a single `<Line>` with `points={}` mutated (Konva reuses the same element on points-set without re-running the layer's `add()` cost).

## Where custom-canvas might help (deferred to v0.3)

This round's measurement doesn't include a custom-canvas comparison, but we expect a custom-canvas implementation to win on:

- **Polygon-fill alpha**: Konva's `closed=true` + 0.2-alpha fill currently uses `globalAlpha` round-trip per frame; HTMLCanvas2D can fold this into a single `ctx.fillStyle = 'rgba(...)'` call.
- **Static-region redraw avoidance**: Konva's `layer.draw()` redraws the entire layer background; Canvas2D's `clearRect` + selective redraw can skip unchanged regions.
- **Annotation rasterization**: Konva issues `getImageData`/`putImageData` only in specific contexts; Canvas2D gives the ground-truth path without library overhead.

What custom-canvas is unlikely to win on:

- Hit-testing (Konva's `getIntersection()` is O(annotations × log) at best; Canvas2D `isPointInPath` is O(n) per query).
- Event plumbing (Konva's Stage event model + per-shape `onContextMenu` saves ~150 lines vs. writing custom mouse hit-testing).
- Visibility / portability on WSL/Raspberry Pi 4 — Konva ships as a tested-dependency, custom-canvas requires us to ship a tested Canvas2D contract.

**Recommendation**: do NOT switch rendering paths until we observe a real-world 2× regression vs. these baseline numbers. Actionable signal is 60fps-on-Wayland drag-create on a synthetic 4096×4096 image; that experiment is deferred to v0.3.
