# Designing a Standards-Compliant Multimedia Annotation Engine for Open-Source QDA Software

This report provides a comprehensive technical analysis and architectural blueprint for integrating image and multimedia annotation features into a v2+ version of an open-source qualitative data analysis (QDA) application. The research focuses on establishing a robust, interoperable foundation aligned with the REFI-QDA .qdpx standard, informed by publicly available documentation and analyses of existing QDA tools. The scope covers six critical dimensions: the implementation patterns of leading QDA software for image coding; the evaluation of JavaScript canvas libraries for a React/Electron frontend; the design of a specialized SQLite data model for image annotations; the architecture for audio/video storage, annotation, and visualization; a future-proof model for transcript synchronization; and a detailed mapping of REFI-QDA schema elements for multimedia content. This work serves as a foundational guide for the development team, ensuring that early architectural decisions support long-term goals of functionality, performance, and data portability without unnecessary complexity.

## Comparative Analysis of Existing Image Coding Implementations

An analysis of how established QDA software implements image region coding reveals critical insights into user expectations, technical approaches, and, most importantly, the practical realities of data interchange via the REFI-QDA standard. While proprietary file formats remain opaque, public documentation and community-sourced knowledge about REFI-QDA exports provide a clear picture of industry practices. The consensus across commercial platforms like NVivo, ATLAS.ti, and MAXQDA is that they support a rich set of interaction paradigms for annotating images, including photographs, maps, and archival documents [[63](https://www.researchgate.net/publication/391149438_RESEARCH_METHODS_AND_STATISTICS_FOR_21ST_CENTURY_EDUCATORS_AND_RESEARCHERS), [65](https://www.academia.edu/128953216/The_Sage_Encyclopedia_of_Qualitative_Research_Methods)]. These tools enable researchers to draw both simple geometric shapes, such as rectangles (bounding boxes), and more complex freehand polygons directly onto source images [[64](https://www.researchgate.net/topic/Research-Tools)]. This functional capability is central to their value proposition, allowing for nuanced coding of spatial information. For instance, ATLAS.ti's extensive documentation on handling multimedia data implies a sophisticated user interface for manipulating objects on top of video and audio files, a principle that extends to static images [[29](https://doc.atlasti.com/ManualWin.v9/ATLAS.ti_ManualWin.v9.pdf), [32](https://manuals.atlasti.com/Win/en/manual/ATLAS.ti_ManualWin.pdf)]. Similarly, MAXQDA is noted for its ability to manage various data types, including images, alongside text and survey data, suggesting a comprehensive approach to data coding [[77](https://www.academia.edu/39694110/Qualitative_Analysis_Software_ATLAS_ti_Ethnograph_MAXQDA_NVivo_)].

The primary divergence among these tools lies not in their capacity to draw regions, but in how they represent and store this geometric data. The choice of coordinate system—whether absolute pixel coordinates or proportional percentage coordinates—is a fundamental architectural decision with significant implications for data portability and reproducibility. Publicly available information strongly indicates that the de facto standard for interchange, driven by the REFI-QDA specification, is the use of a proportional coordinate system [[4](https://www.researchgate.net/publication/341831836_What_is_the_REFI-QDA_Standard_Experimenting_With_the_Transfer_of_Analyzed_Research_Projects_Between_QDA_Software)]. The REFI-QDA XSD schema defines the `AreaReference` element, which is used to describe a selected area on an image, using `left`, `top`, `right`, and `bottom` attributes. These attributes are defined with the type `PercentageType`, meaning their values are proportions of the image's full width and height [[4](https://www.researchgate.net/publication/341831836_What_is_the_REFI-QDA_Standard_Experimenting_With_the_Transfer_of_Analyzed_Research_Projects_Between_QDA_Software)]. This makes an absolute pixel-based coordinate system a poor choice for a canonical storage format, as it would require conversion for any REFI-QDA export, introducing potential rounding errors and breaking direct compatibility with other compliant applications. Therefore, aligning with the percentage-based system is essential for true interoperability.

Perhaps the most crucial insight gleaned from analyzing vendor behaviors is the documented practice of down-conversion during data exchange. The REFI-QDA standard itself represents image regions as AreaReference elements using percentage coordinates, a bounding-box-only model [[4](https://www.researchgate.net/publication/341831836_What_is_the_REFI-QDA_Standard_Experimenting_With_the_Transfer_of_Analyzed_Research_Projects_Between_QDA_Software)]. This limitation is mirrored in the real-world behavior of commercial software. MAXQDA's own official documentation explicitly states that certain project elements, such as paraphrases, are dropped when a project is saved in the REFI-QDA format [[3](https://atlasti.com/features/project-export-in-qdpx-format)]. This serves as a powerful precedent for the proposed application's architecture. It validates a strategy where the native application can store and render rich, arbitrary polygonal geometries for superior user experience, while adhering strictly to the simpler bounding-box model required for REFI-QDA export. This approach is pragmatic and transparent: users who create complex, freehand regions would be clearly informed at export time that this advanced data will be converted to a simple rectangle, much like MAXQDA handles paraphrases. This avoids the immense complexity and risk of creating a custom, unstandardized extension to the REFI-QDA schema, while still serving the user's need for precise annotation within the native application. The open-source tool QualCoder offers a valuable, albeit less documented, example. Its wiki notes on REFI-QDA import/export suggest a focus on fidelity, making it a useful benchmark for understanding the challenges and nuances of working with the standard [[25](https://www.scribd.com/document/740220564/QualCoder)]. Its open-source nature also means its implementation could serve as a practical blueprint for building the audio/video feature pipeline, providing concrete code examples for media handling and annotation linkage [[26](https://www.scribd.com/document/922064613/Manual-Which-Coder)].

In summary, the evidence points toward a two-pronged strategy. The application should provide a rich user interface for drawing various shapes, reflecting user expectations shaped by market leaders. However, its core data model must prioritize a proportional coordinate system to ensure seamless interoperability. The most effective way to achieve this is by adopting a dual-format storage model: using a simplified, standardized format for interchange (bounding boxes in percent coordinates) as the canonical export representation, while storing richer, more precise geometric data (polygons, freehand paths) in a separate, internal extension table. This balances the demands of user functionality with the non-negotiable constraints of the REFI-QDA standard.

## Selection of a Canvas-Based Annotation Library for React/Electron

The selection of a JavaScript canvas library is a critical decision that will directly impact the performance, maintainability, and user experience of the image annotation feature. The ideal library must offer a compelling combination of features tailored to the specific needs of a QDA application: robust support for drawing multiple region types (rectangles, polygons, freehand paths), seamless integration with a React component lifecycle, minimal performance overhead even with a high density of annotations, and explicit support for managing overlapping shapes with distinct visual properties. Based on a comparative analysis of Fabric.js, Konva.js, and other relevant libraries identified in the `image-annotation` GitHub topic, Konva.js emerges as the most suitable choice for this project.

Konva.js distinguishes itself through several key advantages that align perfectly with the project's requirements. First and foremost is its excellent React integration via the `react-konva` package [[16](https://www.npmjs.com/package/react-konva)]. Unlike some other libraries that require treating the canvas instance as an "uncontrolled" component, which can complicate state management in a large React application, `react-konva` provides declarative, reactive bindings to the Konva framework [[16](https://www.npmjs.com/package/react-konva), [55](https://stackoverflow.com/questions/37565041/how-can-i-use-fabric-js-with-react)]. This allows developers to manage the entire state of the annotation canvas using React's familiar patterns, leading to cleaner, more predictable code. Second, Konva's architecture is explicitly designed for performance. It utilizes a multi-layer system where each `Layer` is tied to its own `<canvas>` element [[12](https://konvajs.org/api/Konva.Layer.html)]. This enables a powerful optimization pattern: static background elements (like the image itself) can be drawn once on a bottom layer, while all interactive, draggable, and selectable annotation shapes can reside on a separate, top-most layer. When only the annotations move or change, only the top layer needs to be redrawn, dramatically improving responsiveness and preventing the jank that occurs when redrawing thousands of shapes on a single canvas [[35](https://konvajs.org/docs/performance/All_Performance_Tips.html), [49](https://konvajs.org/docs/performance/Layer_Management.html)]. Stress tests confirm its ability to handle up to 10,000 shapes smoothly with a dedicated drag layer [[36](https://konvajs.org/docs/sandbox/Drag_and_Drop_Stress_Test.html)].

Most critically, Konva provides first-class support for managing the layered visual hierarchy of annotations, a non-negotiable requirement for QDA coding. In qualitative research, it is common for coded regions to overlap. Each code is typically represented by a region with a unique color or style. Konva handles this elegantly through its `zIndex` property. Contrary to CSS `z-index`, Konva's `zIndex` is simply the index of a node within its parent group's children array [[33](https://konvajs.org/docs/groups_and_layers/zIndex.html), [34](https://stackoverflow.com/questions/52754041/how-to-set-zindex-in-konva-js)]. By default, nodes are rendered in the order they are added to the container, so a node added later will appear on top of one added earlier. This provides a natural mechanism for controlling which coded region is visually dominant at an intersection point. Developers can programmatically reorder nodes or use the provided methods like `moveToTop()` and `moveToBottom()` to manage the stacking order dynamically [[37](https://konvajs.org/docs/groups_and_layers/Layering.html)]. This direct control over layering is essential for accurately representing the semantic relationships between intersecting codes. Furthermore, Konva has strong capabilities for rendering labels, tooltips, and other annotation-related text, which can be invaluable for adding context to coded regions [[17](https://konvajs.org/docs/shapes/Label.html)]. The library is actively maintained, with recent commits and ongoing discussions around performance improvements, indicating a healthy and sustainable ecosystem [[36](https://konvajs.org/docs/sandbox/Drag_and_Drop_Stress_Test.html)].

While Fabric.js is a mature and powerful alternative, it presents some trade-offs [[61](https://www.npmjs.com/package/@digs-com/fabric)]. It is an interactive object model built on top of the HTML5 canvas, providing a rich set of features and customization options for controls [[9](https://stackoverflow.com/questions/16027651/bounding-box-appearance-controls-customization-with-fabricjs), [61](https://www.npmjs.com/package/@digs-com/fabric)]. There are even examples of complete image editors built with it, demonstrating its power [[68](https://www.npmjs.com/package/@anu3ev/fabric-image-editor)]. However, its integration with React is often described as less declarative, requiring developers to manage the Fabric instance as an external state [[55](https://stackoverflow.com/questions/37565041/how-can-i-use-fabric-js-with-react)]. This can lead to more boilerplate code and potential pitfalls in state synchronization compared to the more idiomatic `react-konva` approach. Although active, the development momentum appears to favor Konva's modern, component-based philosophy.

Label Studio is another option, but it is a full-fledged, opinionated data labeling platform rather than a general-purpose canvas library [[18](https://developers.redhat.com/articles/2024/05/02/prepare-and-label-custom-datasets-label-studio), [19](https://www.linkedin.com/pulse/unleash-power-data-labeling-label-studio-machine-learning-reply-de-1tapf)]. While it uses libraries like `react-image-annotate` under the hood for its own annotation components, it is not designed to be embedded as a flexible, customizable module within another application [[18](https://developers.redhat.com/articles/2024/05/02/prepare-and-label-custom-datasets-label-studio)]. Attempting to use Label Studio directly would likely introduce excessive overhead and lack the fine-grained control needed for a tightly integrated QDA feature. Other libraries found in the `image-annotation` GitHub topic exist, but none demonstrate the same combination of active maintenance, excellent React integration, and specialized features for managing complex, layered vector graphics as Konva.js.

Its commitment to performance optimization and its active community make it the lowest-risk, highest-reward choice for building a scalable and responsive image annotation tool.

## A Dual-Format SQLite Data Model for Image Annotations

Designing a robust and forward-looking data model for image annotations requires a careful balance between strict adherence to the REFI-QDA standard for interoperability and the flexibility to store richer, more precise geometric data for use within the native application. The most effective solution is a dual-format approach, leveraging SQLite's relational capabilities to store canonical interchange data in a primary table while preserving extended geometric information in a dedicated companion table. This architecture ensures lossless import and storage of user-created annotations while making the limitations of the export standard transparent and manageable.

The cornerstone of this model is the use of a proportional coordinate system as the canonical storage unit. As established by the REFI-QDA specification, which mandates `left`, `top`, `right`, and `bottom` attributes expressed as percentages for image `AreaReference` elements, this is the only viable format for ensuring true data portability [[4](https://www.researchgate.net/publication/341831836_What_is_the_REFI-QDA_Standard_Experimenting_With_the_Transfer_of_Analyzed_Research_Projects_Between_QDA_Software)]. Storing coordinates as percentages makes annotations independent of the display resolution, window size, or physical dimensions of the monitor on which they are viewed. This is a critical feature for qualitative research, where reproducibility and consistency across different environments are paramount. An annotation created on a high-DPI screen should appear identically on a standard-definition laptop. Using absolute pixel coordinates would break this guarantee and create significant hurdles for anyone trying to share projects between different machines or software versions.

For the main table, let's call it `image_selection`, the schema should be designed for maximum clarity and direct mapping to the REFI-QDA standard. It would contain the essential metadata for an annotation: a foreign key linking to the image asset, a foreign key to the assigned code, and the timestamp of creation. Crucially, it would also include four floating-point number columns to define the region as a bounding box: `region_left_percent`, `region_top_percent`, `region_width_percent`, and `region_height_percent`. This structure is unambiguous and can be translated directly into the `<AreaReference>` XML element during a REFI-QDA export operation.

To accommodate more complex, user-driven annotations like freehand polygons and irregular shapes, a second, extension table, `image_annotation_geometry`, is necessary. This table would have a one-to-one relationship with the `image_selection` table via its primary key. Instead of overloading the main table with ambiguous or incomplete data, this extension table would store the richer geometric definition in a standardized, machine-readable format. For polygons and freehand paths, the most practical and efficient format is a JSON array [[27](https://stackoverflow.com/questions/53767311/store-polygon-type-of-data-on-the-sqlite-database)]. Specifically, a polygon could be stored as a JSON array of `[x, y]` pairs, where both `x` and `y` are again expressed as percentages relative to the image dimensions [[47](https://stackoverflow.com/questions/55100786/what-format-of-image-annotation-is-this)]. This approach is human-readable, easy to parse and generate in JavaScript, and can be queried if needed. It strikes an ideal balance between simplicity and expressive power. While other formats like SVG path strings or GeoJSON exist, the JSON array is less verbose and far simpler to manipulate programmatically for the purpose of drawing and editing shapes [[27](https://stackoverflow.com/questions/53767311/store-polygon-type-of-data-on-the-sqlite-database), [45](https://docs.ultralytics.com/usage/simple-utilities)]. For example, modifying a vertex of a polygon involves simply updating a single `[x, y]` pair in the array, a straightforward operation.

The application's logic would operate on this dual-table structure seamlessly. When rendering an annotation, the application would first query the `image_selection` table for the basic data. Then, it would attempt to join with the `image_annotation_geometry` table. If a matching geometry record is found, it would use that rich data to draw the precise shape (e.g., a multi-point polygon). If no geometry record exists, it would fall back to drawing the simpler bounding box defined in the main table. This provides a graceful degradation path and ensures that even annotations imported from a REFI-QDA file (which would only populate the main table) are still displayed correctly. The reverse process applies to export. When generating a `.qdpx` file, the export routine would only ever read from the `image_selection` table, constructing the XML based solely on the percentage-based bounding box data. The existence of the `image_annotation_geometry` table would be entirely ignored during this process. This enforces the policy of down-conversion in a clean, uncomplicated manner.

This hybrid model also aligns well with database best practices. It normalizes the data, separating the core selection metadata from the specialized geometric details. It keeps the primary table lean and focused, which is beneficial for queries that might not need to load the potentially large and complex geometry data (e.g., listing all coded regions for a project). Finally, it provides a clear and honest user experience. The application can inform the user that advanced region types are supported internally, but that REFI-QDA export will convert them to simple rectangles. This transparency builds trust and manages user expectations, avoiding the confusion that arises when complex data is silently dropped or mangled during an export operation. This approach, validated by the documented practices of other tools like MAXQDA, represents a pragmatic and robust solution for the application's data modeling needs [[3](https://atlasti.com/features/project-export-in-qdpx-format)].

## Audio/Video Annotation Architecture and Visualization

The architecture for audio/video annotation must be modular, scalable, and deeply integrated with the core principles of the QDA application, particularly the concept of linking timestamps to codes and, where available, to transcript segments. Drawing inspiration from the open-source QualCoder project provides a solid, practical foundation for this task, while the selection of a modern waveform visualization library is critical for delivering an effective and intuitive user experience. The overall design should prioritize a separation of concerns between media asset management, annotation metadata storage, and visual presentation.

First, the storage of media assets must be handled efficiently. The project's existing folder model, which stores all assets in a dedicated `/assets/` subdirectory, is a sound approach [[52](https://stackoverflow.com/questions/29113494/download-image-to-sqlite-database-with-python)]. The database should never store the large binary media files themselves; instead, it should store only relative file paths (e.g., `assets/audio/recording_01.mp3`). Storing blobs in an SQLite database for large files like audio and video is generally discouraged due to potential performance degradation, increased database size, and complications with backups and replication [[54](https://stackoverflow.com/questions/808499/does-it-matter-what-extension-is-used-for-sqlite-database-files)]. The application's logic would then construct the full local path by combining the project's root directory with the stored relative path.

Second, the metadata for annotations must be structured to support all required functionalities. A new table, `audio_video_selection`, would be created to hold this information. This table would contain fields for `start_ms` and `end_ms` to capture the timestamp range of the selection. It would also include foreign keys linking to the `code` table (to identify the assigned code) and to the media asset being annotated. This structure is analogous to the `Selection` table recommended in previous architectural planning and creates a consistent mental model for the developer and user alike. When a researcher selects a portion of a video and assigns a code, a new row is inserted into this table, precisely defining the temporal scope of that particular piece of coded data.

Third, the integration with transcripts is a key differentiator for a modern QDA tool. This is where the architecture must be designed to handle structured data from the outset. The planned use of OpenAI's Whisper ASR service is a significant driver for this, as Whisper can output word-level timestamps in standard formats like WebVTT and SRT [[20](https://community.openai.com/t/word-level-timestamps-from-whisper-v3s-json-is-invalid/641735), [84](https://community.openai.com/t/how-to-get-whispers-api-to-add-timestamps-to-the-transcripts/501788)]. To leverage this, the application's import pipeline must be capable of parsing these formats. A parser for WebVTT (Web Video Text Tracks Format) or SRT (SubRip Subtitle) can be implemented using existing Node.js libraries such as `node-webvtt` or `subtitle` [[71](https://www.npmjs.com/package/webvtt-parser), [85](https://www.npmjs.com/package/subtitle)]. These parsers would take the structured text file and extract the time ranges and corresponding text, populating a `transcript_segment` table. This table would store the text of each segment along with its own `start_ms` and `end_ms`. With both the `audio_video_selection` table (linking codes to media timestamps) and the `transcript_segment` table (linking text to transcript timestamps), the synchronization between a coded audio clip and its corresponding text becomes a matter of querying for overlapping time ranges. Selecting a coded region in the waveform would trigger a query to find all transcript segments whose time ranges intersect with the selection's start and end times, and highlighting those segments in the transcript view would reveal the associated coded audio.

Finally, the visualization of the waveform is the user's primary interface for this feature. Two prominent JavaScript libraries, Wavesurfer.js and Peaks.js, stand out in the "Best of JS" lists for audio visualization [[2](https://bestofjs.org/projects?tags=audio&sort=yearly), [79](https://bestofjs.org/projects/wavesurferjs)]. Both are powerful, but for this specific use case, Peaks.js appears to be the more promising candidate. While Wavesurfer.js is highly popular and has good React wrappers, some developers have reported difficulties with these wrappers, which can complicate implementation [[78](https://stackoverflow.com/questions/44813585/wavesurfer-js-is-working-fine-but-react-wavesurfer-has-issues)]. More importantly, the documentation and examples for Peaks.js highlight its modularity and its ability to customize the appearance of the waveform, including rendering peaks in different colors based on time ranges [[1](https://stackoverflow.com/questions/69194692/how-to-generate-a-wavesurfer-with-multiple-peak-colors-using-videojs-wavesurfer), [39](https://bestofjs.org/projects/peaksjs)]. This multi-color rendering capability is essential for visually distinguishing coded audio segments from uncoded ones directly on the waveform, providing immediate visual feedback to the user. This feature directly addresses the core requirement of overlaying coded regions onto the audio visualization. Examining the source code of QualCoder for its audio playback and annotation mechanisms would provide a valuable, real-world implementation guide, offering proven patterns for handling media events, tracking time, and synchronizing the UI [[25](https://www.scribd.com/document/740220564/QualCoder)].

## A Structured Transcript Synchronization Model

A robust and flexible transcript synchronization model is a cornerstone of a modern QDA application, enabling deep, multi-modal analysis of spoken data. The architecture for this feature must be designed from the ground up to ingest and normalize structured transcript formats, rather than treating all transcripts as monolithic plain text. This forward-thinking approach is not merely an optimization; it is a strategic imperative driven by the planned integration of Automatic Speech Recognition (ASR) technologies like OpenAI's Whisper, which naturally produce rich, time-aligned output [[22](https://stackoverflow.com/questions/73822353/how-can-i-get-word-level-timestamps-in-openais-whisper-asr)]. Discarding this structure would represent a significant architectural misstep, forcing costly re-engineering in the future and limiting the application's analytical capabilities.

The most effective internal data model is a unified representation of the transcript as a sequence of discrete segments, each containing a word or phrase along with its precise temporal boundaries. This can be implemented in a `transcript_segment` table within the SQLite database. Each row in this table would represent a single unit of speech and would contain at least three pieces of information: the `text` of the segment, and two timestamps, `start_ms` and `end_ms`, marking the beginning and end of that segment within the source audio/video file. This `(word, start_ms, end_ms)` structure is the fundamental atomic unit of the synchronization model. Its power lies in its format-agnosticism. Whether the transcript data originates from a WebVTT file, an SRT file, an OHMS XML document, or a direct API call to Whisper, the final stored representation remains identical [[20](https://community.openai.com/t/word-level-timestamps-from-whisper-v3s-json-is-invalid/641735), [23](https://www.w3.org/TR/webvtt1/), [71](https://www.npmjs.com/package/webvtt-parser)].

This unified model provides immense flexibility and opens the door to a wide range of analytical functions. For example, it allows for precise searching not just of text, but of text within a specific time frame of an audio recording. It enables the automatic generation of short, coded clips directly from highlighted text spans in the transcript. Most importantly, it creates a reliable bridge between the auditory and textual representations of the data. When a user selects a coded region in the waveform visualization, the application can query the `transcript_segment` table to find all segments whose time ranges overlap with the selection, instantly highlighting the corresponding words in the transcript. Conversely, when a user clicks on a specific word or sentence in the transcript, the application can retrieve the `start_ms` and `end_ms` for that segment and scroll and highlight the corresponding area in the waveform and play the associated audio.

Implementing this model requires building parsers for the various input formats. For WebVTT and SRT, which are relatively simple text-based formats, this is straightforward. Libraries like `node-webvtt` provide classes to parse these files and extract the cues (time ranges and text) [[71](https://www.npmjs.com/package/webvtt-parser)]. For OHMS XML, a more complex, domain-specific format used in oral history archives, the parser would need to be more sophisticated to handle additional metadata such as speaker identifiers, subject headings, and geographic information, though the core time-aligned text extraction remains the same [[23](https://www.w3.org/TR/webvtt1/)]. The planned use of Whisper ASR is a perfect fit for this architecture. When calling the Whisper API with the `word_timestamps=True` parameter, the service returns a JSON object that includes word-level timing information, which can be processed directly into the `transcript_segment` table structure [[22](https://stackoverflow.com/questions/73822353/how-can-i-get-word-level-timestamps-in-openais-whisper-asr)]. Some edge cases may arise, such as discrepancies where the word-level segmentation does not perfectly match the generated transcript, but these can be handled gracefully by the ingestion logic [[73](https://community.openai.com/t/timestamp-granularities-word-does-not-match-generated-transcript/733182)]. By committing to this structured-first approach, the application will be able to fully leverage the rich outputs of modern ASR systems and build a truly integrated and powerful multimedia analysis environment.

## REFI-QDA Multimedia Schema Mapping and Compliance

Ensuring compliance with the REFI-QDA standard is the ultimate goal for the data interchange capabilities of the application. This requires a precise mapping of the application's internal data structures to the specific XML elements and attributes defined in the REFI-QDA schema. The standard provides a formal definition for representing multimedia sources and their associated annotations, and adhering to this definition is non-negotiable for successful import and export of projects. The `Projects.xsd` file serves as the authoritative source for this schema, and its elements dictate the structure of the `.qdpx` files [[4](https://www.researchgate.net/publication/341831836_What_is_the_REFI-QDA_Standard_Experimenting_With_the_Transfer_of_Analyzed_Research_Projects_Between_QDA_Software)].

For image annotations, the standard specifies a clear pathway. An image source is defined within a `<Source>` element, which contains a `<Media>` child element pointing to the image asset file. The coding or annotation itself is captured using the `<AreaReference>` element. This element is the key to interoperability. It defines a rectangular region of interest on the image and is populated with four attributes: `left`, `top`, `right`, and `bottom` [[4](https://www.researchgate.net/publication/341831836_What_is_the_REFI-QDA_Standard_Experimenting_With_the_Transfer_of_Analyzed_Research_Projects_Between_QDA_Software)]. Critically, these attributes are all typed as `PercentageType`, confirming that values must be specified as proportions of the image's total width and height (e.g., `left="0.25"` for a region starting a quarter of the way across the image). This reinforces the architectural decision to store all canonical image region data in percentage coordinates. Any attempt to export an image annotation must translate the application's internal geometric representation (even a complex polygon) into a valid `<AreaReference>` with these four percentage-based attributes. The application's export logic must therefore implement a down-conversion function that takes the precise vertices of a polygon and computes the smallest possible bounding box that contains them, then calculates the `left`, `top`, `width`, and `height` as percentages to populate the `AreaReference`.

For audio and video annotations, the standard provides a mechanism for selecting portions of a media file. This is represented by a `<Selection>` element, which can contain a `<TimeRange>`. The `<TimeRange>` element defines a contiguous interval within the media asset using `startTime` and `endTime` attributes. While the exact units are not specified in the provided context, milliseconds are the de facto standard in web and application development for such time intervals. Therefore, the `start_ms` and `end_ms` fields in the proposed SQLite `audio_video_selection` table map directly to these two attributes. The application's export process would simply take the timestamp data from this table and insert it into the XML structure. The inverse process applies to import: when parsing a `.qdpx` file, the application's parser would look for `<Selection>` elements containing a `<TimeRange>` and extract the `startTime` and `endTime` values to create new records in the `audio_video_selection` table.

By meticulously mapping the application's data model to these specific schema elements, the application can guarantee its ability to participate in the broader QDA ecosystem. The internal tables (`image_selection`, `audio_video_selection`, `transcript_segment`) serve as the application's private, flexible data store, optimized for its own functionality. The export routines act as a translator, converting this rich internal state into the simpler, standardized language of REFI-QDA. This disciplined approach to schema mapping, combined with a transparent policy for down-conversion, ensures that the application can both contribute to and benefit from the growing body of research data shared across different QDA platforms.
