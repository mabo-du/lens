# **Analytical Visualization Dashboard for Qualitative Data Analysis: A Technical Implementation Report**

The transition from manual qualitative data coding to Computer-Assisted Qualitative Data Analysis Software (CAQDAS) has fundamentally transformed methodological workflows across the social sciences and humanities.1 While the initial phases of qualitative research inherently rely on close, line-by-line reading and the manual assignment of thematic codes to text spans, subsequent analytical phases require researchers to step back and identify macro-level patterns across the entire corpus.3 Advanced analytical visualisations—such as co-occurrence matrices, code frequency distributions, and semantic network diagrams—facilitate this necessary epistemological transition from granular annotation to theoretical synthesis.5 Developing a free, open-source desktop qualitative data analysis application necessitates the implementation of an analytics layer that matches or exceeds the capabilities of proprietary enterprise tools.  
Implementing a robust analytics layer within a modern React and Electron desktop application introduces significant architectural, computational, and user-experience challenges. The system must rapidly query thousands of character-offset annotations, compute complex statistical coefficients on the fly, and render highly interactive visualisations without blocking the main application thread or triggering excessive memory consumption.7 Furthermore, the user interface must remain intuitively aligned with the epistemological needs of qualitative researchers, prioritizing methodological transparency over algorithmic black boxes.8 This report provides an exhaustive technical implementation plan for the planned v2+ analytics dashboard, covering computational algorithms, rendering strategies, accessibility standards, and desktop application architecture.

## **1\. Co-occurrence Computation Algorithms and Database Architecture**

In the domain of qualitative research, the concept of "code co-occurrence" is not a monolithic, universally agreed-upon metric. Because qualitative data is inherently unstructured and heavily context-dependent, the spatial relationship between two coded segments can carry profound methodological and interpretive implications.9 CAQDAS platforms vary significantly in their operational definitions of co-occurrence, and a robust open-source tool must support multiple computational models to accommodate diverse research designs, ranging from grounded theory to mixed-methods framework analysis.

### **Defining Code Co-occurrence in Qualitative Methodologies**

The operational definition of co-occurrence fundamentally depends on the spatial relationship between character-offset text annotations. Existing enterprise CAQDAS platforms employ distinct computational definitions that a modern open-source tool must emulate and synthesize.  
The most rigid definition of co-occurrence is the exact span or intersection model, wherein two annotations either occupy the exact same character span or overlap to some measurable degree. ATLAS.ti refers to these as spatial relations, categorizing them into discrete operators such as embeddedness, overlapping, and enclosing.10 MAXQDA's Code Relations Browser utilizes a dedicated "Intersection" mode, which restricts the analytical matrix strictly to "real" overlaps where a single data segment is simultaneously coded with multiple distinct codes.11 NVivo handles this intersection requirement via Matrix Coding Queries utilizing the AND boolean operator, which cross-tabulates content coded explicitly to the physical intersection of two qualitative nodes.12 This strict overlap definition is crucial for identifying instances where a participant is discussing two themes simultaneously within the exact same breath or sentence.  
Often, however, qualitative researchers want to know if two concepts are discussed in the same contextual vicinity, even if the exact text spans do not physically intersect. This requires a proximity-based definition. MAXQDA explicitly offers a "Proximity" function that evaluates if two codes are assigned at a definable distance, such as within the same paragraph or within a specific number of rows.11 NVivo supports this contextual analysis via the NEAR operator, allowing users to specify proximity boundaries, such as occurring within *N* words of one another.15 ATLAS.ti evaluates proximity generally but traditionally limits its precise relational querying to segments that are directly embedded or overlapping.17 Proximity queries are computationally more expensive but provide deeper insights into narrative flow, showing how a participant might transition from discussing one theme directly into another.  
The broadest and most computationally forgiving definition of co-occurrence tracks whether two codes appear anywhere within the same document, regardless of their proximity or intersection.11 This document-level co-occurrence is heavily utilized in mixed-methods research to cross-reference thematic prevalence with document attributes or participant demographics.5 For instance, a researcher might want to know if the code for "Financial Anxiety" appears in the same interview transcript as the code for "Career Ambition," even if the themes are discussed hours apart during the interview.

### **The C-Coefficient and Statistical Normalization**

A critical statistical challenge in code co-occurrence analysis is the severe data distortion caused by vastly unequal code application frequencies.19 If a primary code is applied one thousand times across a corpus and a secondary code is applied only ten times, a raw overlap count of five might seem mathematically insignificant in a standard matrix. However, contextually, it means the secondary code overlaps with the primary code in fifty percent of its total applications, representing a highly significant qualitative relationship.  
To systematically address this frequency distortion, ATLAS.ti implements the "c-coefficient," a metric derived from quantitative content analysis.20 The c-coefficient provides a normalized measure of the strength of the relationship between two codes, strictly bound between the values of zero and one. The calculation formula is defined as the number of co-occurrences for the two codes divided by the sum of their individual frequencies minus the co-occurrence count. ### **SQL Query Formulation for Annotation Matrices**

Given a standard relational database schema utilized in desktop applications (such as SQLite), comprising an Annotation table with columns for document\_id, code\_id, start\_char, and end\_char, computing a pairwise co-occurrence matrix requires a highly optimized self-join. To calculate an overlap-based co-occurrence matrix that identifies "real" intersections, the SQL query must locate instances where the character spans logically intersect. The mathematical logic dictates that two spans overlap if the start character of the first span is strictly less than the end character of the second span, and the end character of the first span is strictly greater than the start character of the second span.  
The following SQL statement utilizes Common Table Expressions to first calculate the baseline code frequencies required for the c-coefficient, then identifies the overlapping spans, and finally aggregates the matrix counts:

SQL  
WITH CodeFrequencies AS (  
    SELECT code\_id, COUNT(\*) as freq  
    FROM Annotation  
    GROUP BY code\_id  
),  
Overlaps AS (  
    SELECT   
        A.code\_id AS code\_a,   
        B.code\_id AS code\_b,  
        A.document\_id  
    FROM Annotation A  
    JOIN Annotation B   
      ON A.document\_id \= B.document\_id  
     AND A.code\_id \< B.code\_id   
     AND A.start\_char \< B.end\_char   
     AND A.end\_char \> B.start\_char  
),  
CooccurrenceCounts AS (  
    SELECT   
        code\_a,   
        code\_b,   
        COUNT(\*) as n12  
    FROM Overlaps  
    GROUP BY code\_a, code\_b  
)  
SELECT   
    C.code\_a,   
    C.code\_b,   
    C.n12 AS raw\_cooccurrence\_count,  
    F1.freq AS code\_a\_freq,  
    F2.freq AS code\_b\_freq,  
    CAST(C.n12 AS FLOAT) / (F1.freq \+ F2.freq \- C.n12) AS c\_coefficient  
FROM CooccurrenceCounts C  
JOIN CodeFrequencies F1 ON C.code\_a \= F1.code\_id  
JOIN CodeFrequencies F2 ON C.code\_b \= F2.code\_id  
ORDER BY c\_coefficient DESC;

The self-join condition utilizing the less-than operator on the code identifiers prevents the generation of duplicate pairs in the matrix and eliminates self-loops where a code is compared against itself.

### **Database Performance Assessment and Scaling**

For a moderate qualitative dataset containing five thousand annotations distributed across one hundred unique codes, the computational complexity of the naive self-join approach operates at ![][image5] per document. Assuming the five thousand annotations are evenly distributed across fifty qualitative documents, the maximum number of comparisons required per document is ten thousand, resulting in roughly half a million comparisons across the entire corpus.  
Modern SQLite database engines, which are heavily utilized as the persistence layer in Electron applications, can execute this specific self-join in a fraction of a second, provided that the database schema is heavily optimized with composite indexes. Creating a composite index on the document and code identifiers, alongside an R-Tree or spatial index on the character boundaries, will drastically reduce the query scan times by allowing the database engine to utilize bounding-box logic for the intersection calculations. For five thousand annotations, the query performance will be virtually instantaneous, rendering in under fifty milliseconds.  
However, if the research corpus scales to encompass five hundred thousand annotations—a reality for massive, team-based framework analysis projects—the ![][image5] nature of the self-join query could become a severe application bottleneck. Executing this query synchronously within the main Electron process would inevitably block the React user interface thread, leading to application freezes and a degraded user experience. To mitigate this, the database computation must be offloaded to a dedicated Node.js Web Worker or a separate hidden Electron renderer process, allowing the query to execute asynchronously while providing the user with a non-blocking loading state.

## **2\. Co-occurrence Matrix Visualisation rendering**

Once the mathematical co-occurrence matrix is computed by the persistence layer, rendering it interactively within the graphical user interface allows researchers to identify thematic clusters visually. A matrix containing one hundred distinct codes yields a potential grid of ten thousand individual cells. The choice of rendering library in a React and Electron desktop environment is heavily dictated by performance scaling constraints, Document Object Model node limits, and complex user interaction requirements.

### **Library Comparison for Heat-map Rendering**

Rendering ten thousand distinct analytical cells introduces immediate and severe performance constraints if the application attempts to utilize standard React rendering via the Document Object Model. Every node injected into the DOM requires the browser engine to calculate layout, paint, and composite operations, which exponentially degrades scrolling and interaction performance.

| Library | Rendering Engine | Est. Bundle Size (Gzipped) | Performance at 100x100 Matrix | Interactivity and UX Features |
| :---- | :---- | :---- | :---- | :---- |
| **Recharts** | SVG / React DOM | \~120 KB | Poor. 10,000 SVG nodes will block the main thread and cause scroll lag. | Excellent React integration, basic tooltips, highly accessible.23 |
| **Plotly.js** | WebGL / SVG | \~3.0+ MB | Excellent. Easily handles millions of points via WebGL acceleration. | Native zoom, pan, and hover, but requires massive bundle overhead.24 |
| **Nivo** | Canvas / SVG | \~285 KB (Core) | Excellent via ResponsiveHeatMapCanvas | Highly polished aesthetics, fluid animations, canvas variant solves DOM limits.25 |
| **Observable Plot** | SVG | \~100 KB | Poor to Moderate. Bound by SVG node limits in the DOM. | Expressive grammar of graphics, but lacks out-of-the-box rich React pan/zoom capabilities.26 |
| **D3.js** | SVG / Canvas | \~93 KB (Core) | Excellent, assuming a custom Canvas implementation is explicitly written. | Provides absolute ultimate control, but requires manual state synchronization, bypassing React.27 |

When evaluating these libraries for an open-source QDA tool, maintaining a small bundle size while guaranteeing performance is paramount. **Plotly.js**, despite its immense analytical power and WebGL capabilities, should generally be avoided for this specific component due to its massive gzipped bundle size exceeding three megabytes.24 Including Plotly significantly bloats the application memory footprint. The plotly.js npm package and its React wrapper react-plotly.js (github.com/plotly/react-plotly.js) are powerful but overly heavy for a targeted dashboard.  
**Recharts** (recharts), while incredibly popular, lightweight, and modular, relies exclusively on SVG rendering.28 Attempting to render a one hundred by one hundred matrix in Recharts requires injecting at least ten thousand \<rect\> nodes into the React Virtual DOM.25 This architecture will severely degrade scroll performance and cause noticeable input latency. **Observable Plot** (@observablehq/plot), though celebrated for its elegant grammar of graphics, faces the exact same SVG DOM-bound limitations and requires additional wrapper code to function idiomatically within React components.26  
**D3.js** (d3) offers the granular control necessary to build a highly performant HTML5 Canvas implementation from scratch.27 However, integrating D3.js with React requires intentionally bypassing React's declarative rendering model. The developer must manually mutate the DOM or Canvas context within useLayoutEffect hooks, which dramatically increases the maintenance burden and code complexity of the open-source project.29  
Consequently, **Nivo** (@nivo/core, @nivo/heatmap) emerges as the optimal compromise between React idiomaticity, aesthetic polish, and raw performance.25 Nivo provides dual rendering variants for its charts, offering both SVG and Canvas implementations. By utilizing the Canvas implementation (\<ResponsiveHeatMapCanvas\>), the application completely avoids injecting ten thousand nodes into the React Virtual DOM.25 The entire matrix is drawn onto a single \<canvas\> element. This architecture guarantees smooth scrolling, instantaneous sorting of matrix rows, and fluid hover tooltips without freezing the Electron browser window, smoothly handling matrices far exceeding the one hundred code threshold.

### **Existing Open-Source Bioinformatics Components**

The bioinformatics and computational biology sectors frequently deal with massively multidimensional data, such as gene expression arrays, and have subsequently developed several open-source heatmap components that can be conceptually adapted for qualitative data analysis.  
For instance, **Clustergrammer**, developed by the Ma'ayan Lab and available open-source on GitHub (github.com/MaayanLab/clustergrammer), provides highly interactive heatmaps designed specifically for dynamically sorting rows and columns to reveal hidden data clusters.30 Another robust open-source tool, **MSEABOARD** (github.com/lucanest/mseaboard), offers interactive distance matrix visualizations explicitly utilizing React.31 While these bioinformatics tools are highly specialized for structural biology and phylogenetic trees, inspecting their source code provides invaluable architectural patterns for implementing canvas-based semantic zooming, panning, and hierarchical clustering of matrix rows and columns within a React context. Adapting the hierarchical sorting logic found in these repositories would allow the QDA application to automatically cluster codes that frequently co-occur, visually grouping theoretical concepts without requiring manual researcher intervention.

## **3\. Code Frequency Charts and Methodological Accessibility**

Code frequency charts serve to answer fundamental quantitative questions regarding thematic prevalence and distribution across the qualitative corpus. While these charts are conceptually straightforward, their visual execution must adhere to strict accessibility guidelines. Data visualization is fundamentally a form of communication, and every visual element must explicitly serve human understanding.32

### **Essential Chart Types in Qualitative Workflows**

Based on published qualitative researcher workflows, three distinct frequency chart variations are indispensable for a comprehensive analytics dashboard:  
The first is the simple bar chart of code application counts. This chart is used to display the absolute frequency of codes across the entire project. To maximize utility, these charts must be dynamically sortable, allowing researchers to toggle between frequency-based sorting to identify dominant themes, and alphabetical sorting to locate specific concepts quickly.33  
The second essential visualization is the stacked bar chart, which displays code frequency segmented by document or participant attribute. These charts are crucial for comparative cross-case analysis.5 By stacking code frequencies per document, researchers can immediately detect variance in the data. For example, a researcher can visually identify if a specific theme is heavily concentrated within a single anomalous interview or if it is distributed evenly and consistently across the entire participant pool.  
The third critical visualization is the timeline or distribution view. Qualitative research is heavily invested in narrative flow and the evolution of concepts. A timeline view, often rendered as a scatter plot, rug plot, or area chart, illustrates exactly where codes appear sequentially within a single document or across a temporally ordered corpus.5 Plotting codes by their character-offset position within a document allows researchers to track narrative shifts, such as how a participant's discussion moves from historical context at the beginning of an interview to future anxieties toward the end. Plotting codes by document import date is equally valuable for longitudinal studies, showing how thematic prevalence shifts over months or years of ethnographic observation.

### **Accessibility and Color-Blind Friendly Design**

A major failure in standard data visualization is the reliance on default categorical color palettes, specifically standard red and green spectrums.34 These palettes render charts entirely illegible to users with Deuteranomaly and Protanomaly, the most common forms of red-green color blindness, which affect approximately eight percent of the male population globally.36  
A rigorous accessibility implementation in the QDA dashboard requires the intentional rejection of "stoplight" palettes.35 Instead, the application must utilize perceptually uniform and color-blind safe palettes, such as the Viridis color scale or dedicated ColorBrewer categorical palettes, which maintain distinct luminance steps.33 Furthermore, accessibility guidelines dictate redundant data encoding. The interface must never rely on color alone to convey meaning.32 Charting libraries must support the injection of patterns, such as hatching or stippling, into chart elements, or utilize distinct geometric shapes for scatter plot markers to ensure differentiation without color.32  
Beyond visual encoding, the charts must be navigable via keyboard and accessible to screen readers. For standard bar and stacked bar charts, **Recharts** excels in accessibility.37 It natively provides robust keyboard navigation out of the box, allowing visually impaired researchers to tab through individual data points and trigger screen-reader-compatible tooltips.37 While Nivo is superior for massive canvases like the co-occurrence matrix, Recharts (recharts) is highly recommended for the frequency bar charts due to its lightweight SVG structure, modularity, and accessibility-first architecture.24

## **4\. Network and Relationship Diagrams for Semantic Mapping**

Force-directed graphs represent qualitative codes as interconnected nodes and their co-occurrence frequencies as weighted edges. These network diagrams allow researchers to step out of tabular matrix constraints and perceive their entire conceptual framework as an organic semantic network.38 Scaling these graphs introduces intense computational physics challenges, as the underlying layout algorithm must continuously calculate repulsion forces between all nodes and attraction forces along all edges on every animation frame until the graph stabilizes.

### **Comparison of Network Visualisation Libraries**

Evaluating rendering libraries for a graph containing up to five hundred concept nodes and potentially five thousand co-occurrence edges requires a deep analysis of underlying rendering engines—specifically the performance disparities between SVG, Canvas, and WebGL—and the availability of bundled graph theory algorithms.7

| Library | Rendering Engine | Est. Bundle Size | Strengths | Limitations |
| :---- | :---- | :---- | :---- | :---- |
| **react-force-graph** | WebGL (Three.js) | Moderate | Provides native React bindings. GPU-accelerated 2D, 3D, and VR graphs. Extremely performant, easily handling thousands of nodes.39 | Requires external dependencies like Three.js, which can be computationally heavy for simple use cases. |
| **Cytoscape.js** | Canvas / SVG | \< 5 KB (Core SDK) | The absolute bioinformatics standard. Exceptional built-in graph theory algorithms, specifically automated edge bundling.41 | Steep, imperative learning curve. The official React wrapper (react-cytoscapejs) is somewhat dated and less maintained.43 |
| **Sigma.js** | WebGL / Canvas | \< 50 KB | Purpose-built explicitly for massive graphs containing thousands of nodes and edges. Utilizes graphology for rich graph theory support.44 | More complex state management, though the @react-sigma/core module bridges this well for modern development.46 |
| **D3.js (d3-force)** | SVG / Canvas | \~93 KB (Core) | Absolute, granular customization. Easily integrates with standard D3.js analytical workflows.27 | The SVG implementation severely chokes around 2,000 nodes due to DOM limits.7 Requires manual physics loop writing for Canvas. |

For a qualitative data analysis application, the theoretical upper bound of five hundred nodes and five thousand edges pushes the absolute limits of standard SVG rendering, but is easily and smoothly handled by HTML5 Canvas and WebGL contexts.7  
**Cytoscape.js** (cytoscape, github.com/cytoscape/cytoscape.js) presents a highly compelling option due to its specific analytical features inherited from the bioinformatics sector. In highly dense qualitative code networks, rendering five thousand edges simultaneously creates severe visual clutter, often resulting in an unreadable "hairball." Cytoscape.js provides built-in *Edge Bundling* algorithms.42 This graph theory technique visually merges parallel and adjacent edges into tight, curved bundles, vastly improving the readability of complex thematic relationships. However, bridging its imperative, vanilla JavaScript API with React's declarative state requires significant boilerplate code, and the existing react-cytoscapejs wrapper has not seen major updates in several years.43  
**Sigma.js** (sigma, github.com/jacomyal/sigma.js) is an exceptionally powerful alternative designed specifically for drawing massive graphs. Combined with its companion library graphology 45, Sigma.js handles thousands of nodes and edges flawlessly. The modern @react-sigma/core package provides excellent React lifecycle integration.46  
However, **react-force-graph** (react-force-graph-2d, github.com/vasturiano/react-force-graph) provides the fastest time-to-value for the development roadmap.39 It abstracts the complex WebGL and Three.js rendering pipelines into highly declarative React components. It easily handles the target of five thousand edges without dropping frames.40 Crucial user experience interactions—such as dragging nodes to organize concepts, semantic zooming, and click-to-highlight functionality that isolates connected sub-graphs—are supported natively via component props.  
For the v2+ roadmap, utilizing react-force-graph-2d is highly recommended for the primary semantic network view. It provides the necessary GPU acceleration out-of-the-box and aligns seamlessly with React's component lifecycle, minimizing development overhead while ensuring robust, fluid performance for researchers analyzing complex conceptual webs.40

## **5\. Methodological Efficacy: What Analytical Views Do Researchers Actually Use?**

Implementing advanced visualizations carries the inherent risk of software bloat, where developers build complex graphical tools that look impressive but offer little actual methodological utility. An evidence-based software development approach dictates that prioritization should focus on features that qualitative researchers demonstrably use and value, while explicitly deprioritizing visual tools considered superficial or methodologically flawed.2

### **High-Value Analytical Views**

Published usability studies, methodological literature, and analyses from organizations like the CAQDAS Networking Project at the University of Surrey (surrey.ac.uk/computer-assisted-qualitative-data-analysis) reveal that qualitative researchers heavily rely on tools that facilitate systematic cross-case analysis and rigorous thematic comparison.50  
The undisputed core of advanced qualitative analytics is the matrix query, known in NVivo as the Matrix Coding Query and in MAXQDA as the Code Relations Browser.3 Researchers use these cross-tabulations relentlessly to systematically evaluate how qualitative themes intersect with participant demographics, source attributes, or other overlapping themes.5 Matrices are highly valued because they serve as a bridge rather than an endpoint; they preserve the vital link to the underlying qualitative data. In these tools, double-clicking a quantitative cell in the matrix instantly retrieves the original contextual text segments.53 This preserves the inductive, interpretative nature of qualitative inquiry, ensuring the researcher is never fully disconnected from the participant's voice.  
Furthermore, visualisations that illustrate how codes are distributed within a specific document—such as MAXQDA's Document Portrait—or how they trend sequentially over time are heavily utilized in longitudinal studies and narrative analysis methodologies.5 They allow researchers to comprehend the thematic "shape" or structural progression of an interview at a glance. Semantic networks and concept maps are also actively used for theory building, particularly within Grounded Theory frameworks. Mapping codes as visual nodes and analyzing their spatial proximity helps researchers draft overarching theoretical frameworks and visually audit their own hierarchical coding structures to identify conceptual gaps.38

### **The Methodological Fallacy of the Word Cloud**

Conversely, visual tools based purely on raw word frequency, most notably the **Word Cloud**, are frequently criticized and dismissed in rigorous academic literature as superficial "gimmicks".54 While word clouds offer a rapid, visually engaging summary of raw text, they suffer from severe, fundamental methodological limitations.56  
At best, word clouds represent a crude quantification of qualitative data.57 They present mere numerical counts of decontextualized strings, systematically stripping away meaning, syntax, sentiment, and context.55 In a qualitative research paradigm, the absolute frequency of a word does not equate to its thematic or theoretical importance. A participant might mention a highly critical issue only once, but in a profoundly emotional or analytically significant manner. Word clouds fail entirely to capture this nuance. Furthermore, word frequency tools only count sequences of letters; they cannot differentiate between a word used positively or negatively.  
As noted by qualitative methodologists and highlighted in evaluations by researchers, word clouds are often viewed as a quantitative reduction that can distract inexperienced researchers from the hard work of deep, interpretive analysis.57 Studies comparing manual thematic extraction against computerized word cloud text analysis consistently highlight the limitations of relying on frequency visualizations for meaningful insight.59 While they may be included in software as a basic tool for rapid, high-level exploratory overviews, they should be strictly deprioritized in the v2+ development roadmap. Engineering resources should instead be relentlessly focused on building rigorous, interactive matrix queries and fluid conceptual network graphs that actually advance qualitative theory building. For detailed reviews on CAQDAS usability and feature necessity, the resources provided by the CAQDAS Networking Project (https://www.surrey.ac.uk/computer-assisted-qualitative-data-analysis/resources/choosing-appropriate-caqdas-package) remain the academic gold standard.50

## **6\. Desktop Dashboard Architecture in React and Electron**

Integrating a comprehensive suite of GPU-accelerated force-directed graphs, complex canvas heatmaps, and SVG-based bar charts into a single desktop qualitative data analysis application necessitates meticulous architectural planning. The primary engineering considerations revolve around memory management within the V8 JavaScript engine, the facilitation of multi-monitor workflows for professional researchers, and the absolute requirement for data synchronization and freshness across the entire React component tree.

### **Architectural Approaches and Memory Implications**

There are three primary architectural paradigms for handling complex, memory-intensive analytical views within an Electron desktop environment:  
The first paradigm involves rendering **Inline Panels** within the main application window's component tree. In this architecture, visualisations are rendered directly adjacent to the main code editor and document viewer, managed via standard React routing or tab components. The primary advantage is the simplicity of state management. The visualisations share the exact same global state store (e.g., Redux or Zustand) as the text editor, resulting in zero Inter-Process Communication (IPC) overhead. However, the critical flaw is memory consumption. Large D3.js or WebGL canvas instances consume the same memory pool as the main text editor. Because the V8 engine has hard memory limits, loading a massive network graph alongside a heavily annotated document can cause the browser thread to choke. If the WebGL graph causes a UI freeze or crashes, it takes down the entire application window, destroying the user's unsaved analytical work.  
The second paradigm involves spawning a **Separate Electron BrowserWindow** rendered on demand for each analytical view. The advantage here is true multi-monitor support, highly requested by professional researchers. Furthermore, each BrowserWindow runs in a completely separate Chromium rendering process. This entirely isolates memory usage, meaning a heavy matrix computation cannot lock the main text editor thread. However, this architecture is extremely heavy. Each new BrowserWindow duplicates the base Chromium overhead, often consuming 50 to 100 megabytes of RAM merely to open an empty window. More critically, React state cannot be shared directly across separate OS-level windows. The qualitative data must be serialized, sent via asynchronous IPC messages over the Electron bridge, and deserialized in the new window. This creates a nightmare for maintaining "data freshness." If a researcher modifies a code name or deletes an annotation in the main window, the application must broadcast IPC events to update the charts in the secondary window, often leading to race conditions and out-of-sync visualisations.61  
The third, and optimal, paradigm is a **Detachable Panel System** operating as a tiling window manager within a single Electron window. This provides a multi-panel interface that allows users to drag, drop, split, and tile analytical views akin to a modern Integrated Development Environment (IDE) like Visual Studio Code. This approach maximizes screen real estate while keeping all components strictly within a single React context and a single operating system window. It ensures absolute data freshness without any serialization or IPC overhead.

### **Tiling Window Managers: React Mosaic vs. Golden Layout**

To implement a robust detachable panel system, developers must choose between several layout managers available in the JavaScript ecosystem.  
**Golden Layout** (github.com/golden-layout/golden-layout) is a historically mature, heavily utilized layout manager offering comprehensive drag-and-drop mechanics, popout windows, and docking.61 However, Golden Layout was fundamentally designed in the jQuery era. Its underlying architecture involves aggressive, imperative DOM manipulation completely outside of React's lifecycle. While React wrappers exist for Golden Layout, they frequently suffer from severe memory leaks and state synchronization issues because the layout manager attempts to mount and unmount React components imperatively, fighting against React's Virtual DOM reconciliation engine.  
**React Mosaic** (github.com/nomcopter/react-mosaic) is a React-first tiling window manager created specifically to solve the precise integration issues inherent in Golden Layout.61 It utilizes an idiomatic React component tree and the React Context API, allowing complex child components to flow naturally through the standard React reconciliation cycle. It is highly performant and was explicitly designed for IDE-like exploratory tools where users need to rapidly context-switch between text editors, terminal outputs, and graphical charts.61  
Another highly rated, modern alternative in this space is **RC-Dock** (github.com/ticlo/rc-dock). RC-Dock supports native React docking, allows users to pop out panes, and crucially, implements advanced cache modes to prevent unnecessary component re-rendering when panels are hidden or resized.64 This cache mode is invaluable when dealing with computationally heavy network graphs that should not be recalculated simply because the user adjusted a panel border.

### **Architectural Recommendation for v2+**

The optimal architecture for the QDA analytics dashboard is a single-window detachable panel system utilizing an idiomatic React library such as **React Mosaic** or **RC-Dock**.  
This approach perfectly aligns with the specific epistemological and workflow needs of qualitative researchers, who constantly iterate back and forth between macro-analysis in the visualisations and micro-analysis in the text editor.61 By keeping the analytical dashboard in the exact same Electron renderer process as the main application, all visualisations share the same global state manager natively. If a researcher renames a code or adds a new text annotation, the heatmaps and force-directed graphs automatically and instantaneously re-render via standard React state updates, guaranteeing absolute data freshness without the latency of IPC messaging.  
To rigorously mitigate the memory and performance risks inherent in a single-window, single-thread architecture, the application must adopt defensive programming strategies. The codebase should aggressively utilize React's useMemo hooks for caching the heavy matrix SQL computations. Furthermore, it must implement dedicated Node.js Web Workers for executing the ![][image5] SQL self-joins completely off the main UI thread. Finally, by strictly utilizing optimized Canvas and WebGL libraries (such as Nivo and react-force-graph) instead of DOM-heavy SVG solutions whenever the data exceeds one thousand points, the application will ensure a seamless, highly performant, and methodologically sound analytical environment capable of scaling to enterprise research demands.
