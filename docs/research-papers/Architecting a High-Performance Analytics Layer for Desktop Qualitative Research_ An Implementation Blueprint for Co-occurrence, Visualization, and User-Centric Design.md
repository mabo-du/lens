# Architecting a High-Performance Analytics Layer for Desktop Qualitative Research: An Implementation Blueprint for Co-occurrence, Visualization, and User-Centric Design

## Co-occurrence Computation and Algorithmic Optimization

The foundation of the analytics layer is the algorithm for computing code co-occurrence, a feature central to identifying thematic patterns across a research corpus. The primary implementation target is defined by any-overlap character-span definitions, a standard adopted by leading qualitative data analysis (QDA) software . This approach is favored over exact span matching or proximity-based definitions because it aligns with how researchers intuitively conceptualize codes appearing together in a passage; two annotations co-occur if their textual ranges intersect, regardless of whether they cover the same exact characters or are adjacent . The canonical SQL expression to define this relationship is `start_a < end_b AND start_b < end_a`, which accurately captures all forms of interval overlap, including partial containment and wrapping scenarios [[19](https://www.linkedin.com/posts/sachintukumar_time-overlap-logic-in-sql-activity-7375476706951929856-uLEt), [58](https://www.volcengine.com/article/623988)]. This logical condition forms the basis of the computation, but its naive implementation poses significant performance challenges that must be addressed through sophisticated database optimization.

A straightforward implementation would involve a self-join on the `Annotation` table, where each annotation is joined with every other annotation within the same document to check for overlap [[59](https://www.alibabacloud.com/blog/599145)]. For a corpus with 5,000 annotations, this approach would result in approximately $5000 \times 5000 / 2 = 12,500,000$ comparisons, demonstrating a quadratic time complexity, O(n²) [[60](https://dba.stackexchange.com/questions/127323/performance-of-a-self-join-in-a-temporal-table-with-a-single-days-difference-be)]. Such a computational load is prohibitive and would render the feature unusable during an active coding session, directly contradicting the goal of a performant application . Therefore, achieving acceptable performance hinges entirely on leveraging advanced SQL query optimization techniques provided by modern relational databases like PostgreSQL or SQLite.

The most critical optimization leverages specialized indexing structures. Standard B-Tree indexes on columns like `(document_id, start_char)` are insufficient for this task. While they can quickly locate rows based on a starting character position, they cannot efficiently prune the search space to find all potential overlaps, often resulting in a full table scan and poor performance [[42](https://www.red-gate.com/simple-talk/databases/postgresql/solving-the-overlap-query-problem-in-postgresql/)]. The superior solution is to use a Generalized Search Tree (GiST) index in conjunction with a range data type. By creating a column of type `tsrange` (or similar) from the `start_char` and `end_char` values, the database can utilize highly efficient spatial indexing algorithms. The query can then be written using the overlap operator (e.g., `&&` in PostgreSQL), transforming the problem from one with potentially quadratic performance to one with linear or near-linear complexity [[42](https://www.red-gate.com/simple-talk/databases/postgresql/solving-the-overlap-query-problem-in-postgresql/)]. For example, the optimized query would create a GiST index on a range column, such as `CREATE INDEX ON annotations USING GIST(tsrange(start_char, end_char))`, and then execute a join using the `&&` operator to find all pairs of annotations whose ranges overlap within the same document. This technique dramatically reduces the number of disk I/O operations and CPU cycles required, making the computation feasible for large datasets.

Beyond indexing, further performance tuning can be achieved through manual intervention and query rewriting. Some database systems, like Databricks' Spark runtime, offer specific "range join optimizations" that partition the data into bins based on character position [[18](https://learn.microsoft.com/en-us/azure/databricks/optimizations/range-join)]. By instructing the query optimizer to only consider joins between annotations in overlapping bins, the search space is drastically reduced [[17](https://stackoverflow.com/questions/27433474/efficiently-joining-over-interval-ranges-in-sql), [18](https://learn.microsoft.com/en-us/azure/databricks/optimizations/range-join)]. While the desktop application may not use Spark, the principle can be applied manually by adding constraints to the query. If domain knowledge indicates that no annotation typically exceeds a certain length (e.g., a paragraph), this information can be added to the `WHERE` clause. For instance, if the maximum duration is known to be four hours, the query can be modified to include a constraint like `AND p.enter >= o.end_time - '4 hours'::INTERVAL`, effectively turning the complex overlap join into a more manageable range query that can be served by a standard B-Tree index [[42](https://www.red-gate.com/simple-talk/databases/postgresql/solving-the-overlap-query-problem-in-postgresql/)]. Additionally, rewriting queries from using `IN` clauses with subqueries to direct `INNER JOIN`s can sometimes yield better performance, as the latter is often more easily optimized by the query planner [[71](https://dba.stackexchange.com/questions/288893/optimizing-nested-sqlite-query)].

The following table summarizes the comparison of different SQL approaches for co-occurrence computation:

| Approach | SQL Condition Example | Indexing Strategy | Performance Complexity | Suitability |
| :--- | :--- | :--- | :--- | :--- |
| Naive Self-Join | `SELECT * FROM Annotation a1, Annotation a2 WHERE a1.document_id = a2.document_id AND a1.annotation_id < a2.annotation_id AND a1.start_char < a2.end_char AND a2.start_char < a1.end_char;` | None or standard B-Tree | O(n²) | Unsuitable for >1,000 annotations |
| Optimized Range Join | `SELECT ... FROM annotations a1 JOIN annotations a2 ON a1.document_id = a2.document_id AND tsrange(a1.start_char, a1.end_char) && tsrange(a2.start_char, a2.end_char);` | GiST on `tsrange` column | O(n log n) or better | Recommended for production use |
| Bin-Based Manual Optimization | `... WHERE a1.char_bin = a2.char_bin AND ...` | Composite index on bin and document | O(n) or better | Suitable for databases without native range join support |

In conclusion, the successful implementation of the co-occurrence feature is contingent upon moving beyond a simple self-join query. The architecture must incorporate a robust database schema with a range data type and a corresponding GiST index. This foundational optimization is non-negotiable for ensuring the analytics layer remains responsive even as the volume of coded data grows towards the target of 5,000 annotations. The choice of database engine (e.g., SQLite vs. PostgreSQL) will influence the specific syntax and availability of these advanced features, necessitating thorough benchmarking within the chosen technology stack.

## Interactive Heatmap Visualization for Co-occurrence Matrices

Once the co-occurrence counts have been calculated and stored, the next step is to present them in a visually intuitive format. The co-occurrence matrix, rendered as a heatmap, is a staple in QDA software for revealing patterns of code association [[83](https://pmc.ncbi.nlm.nih.gov/articles/PMC10422407/), [120](https://doc.atlasti.com/ManualWin.v9/ATLAS.ti_ManualWin.v9.pdf)]. The selection of a visualization library for this component in a React/Electron environment involves a careful trade-off between developer productivity, customization capabilities, performance, and accessibility. The primary candidates for evaluation are Recharts, D3.js, Plotly.js, and Observable Plot, each with distinct advantages and disadvantages for this specific use case.

Recharts is a popular choice for developers seeking to build charts declaratively within a React application, as it provides components for various chart types like bar charts and line graphs [[1](https://ably.com/blog/top-react-chart-libraries), [38](https://stackoverflow.com/questions/71403244/how-would-i-go-about-making-a-seaborn-categorical-plot-in-d3-js-or-the-recharts)]. Its main strength lies in its simplicity and ease of integration. However, for a complex, interactive heatmap, Recharts' abstractions become a limitation. Customizing interactions such as row and column reordering, implementing smooth zooming with panning, and designing rich hover tooltips that display detailed statistical information (e.g., raw counts, percentages) requires significant effort or extending the library's core components, defeating the purpose of its simplicity. While functional, it may not deliver the level of bespoke interactivity required for a professional-grade analytics tool.

Plotly.js is a powerful, high-level charting library capable of producing a wide array of over 40 chart types, including 3D charts and scientific graphs [[55](https://plotly.com/javascript/)]. It offers extensive interactivity out of the box, including hover tooltips and built-in zoom/pan functionality. However, its suitability for this project is severely hampered by two major drawbacks. First, its bundle size is exceptionally large, often exceeding 10 MB in minified form [[27](https://community.plotly.com/t/how-can-i-reduce-bundle-size-of-plotly-js-in-react-app/89910), [28](https://community.plotly.com/t/plotly-js-size-is-huge-3mb-in-production-build/45407)]. Given that the application is a desktop tool where download size and initial load time matter, integrating such a heavy dependency is a significant architectural concern. Second, users have reported that Plotly's animations and hover interactions can feel "clunky and choppy," particularly when compared to more lightweight alternatives, which detracts from the fluid user experience expected in a modern application [[20](https://community.plotly.com/t/dash-plotlys-poor-animation-vs-other-js-charting-like-d3-why/54411)]. These factors make it a less-than-ideal choice for the core heatmap visualization.

Observable Plot represents a modern, declarative approach to visualization, emphasizing clarity and composability [[21](https://d3js.org/)]. It excels at creating clean, publication-quality graphics with a concise API. While it is undoubtedly a powerful tool, it may be overkill for the relatively straightforward task of rendering a static or moderately interactive heatmap. Its learning curve and ecosystem maturity are still developing compared to more established libraries, and it might lack some of the pre-built, complex interaction patterns found in D3.js.

This analysis leads to a clear recommendation: **D3.js** is the most appropriate library for rendering the co-occurrence matrix heatmap. Despite its steeper learning curve, D3.js offers unparalleled flexibility and control, allowing the development team to build a truly bespoke visualization that precisely meets the project's requirements [[21](https://d3js.org/)]. With D3.js, it is possible to construct a heatmap using SVG or Canvas elements and attach custom event handlers for every interaction. This enables the creation of a seamless user experience featuring:
*   **Interactive Sorting:** Users can click headers to sort rows and columns by frequency or alphabetically, providing immediate insight into the most and least connected codes.
*   **Rich Hover Tooltips:** On mouseover, a tooltip can display detailed information, including the raw co-occurrence count, percentage of total co-occurrences, and perhaps even a sample of the overlapping text segments.
*   **Smooth Zooming and Panning:** For larger matrices, a zoomable and pannable view allows users to explore dense regions of the heatmap without losing context.

While building the heatmap with D3.js requires more development effort than using a higher-level wrapper like Recharts, the resulting product will be more performant, customizable, and aligned with the project's goals for a polished, professional interface. The final choice should also consider bundle size, as D3.js itself is a substantial library, though tree-shaking can help reduce its impact. The key is to leverage D3's power to create an interactive and insightful visualization rather than relying on the more limited capabilities of simpler charting libraries.

## Accessible and Insightful Code Frequency Charts

Beyond identifying relationships between codes, understanding the frequency and distribution of individual codes is a fundamental aspect of qualitative analysis [[7](https://journals.sagepub.com/doi/10.1177/16094069251384401)]. The analytics layer must provide multiple chart types to support different analytical questions, while adhering to a strict requirement for accessibility. The three essential chart types identified are: a simple bar chart showing total code application counts; a stacked bar chart displaying code frequency per document; and a timeline view illustrating the temporal application of codes . The selection of a React charting library must therefore prioritize not only charting capabilities but also built-in support for accessibility standards like WCAG 2.1 AA.

For the simple bar chart and stacked bar chart, numerous libraries offer strong support. Libraries like Nivo and Visx are noted for their excellent accessibility features and open-source nature, making them strong candidates [[54](https://juejin.cn/post/7633633244707045410)]. Nivo, in particular, is highlighted as an option with comprehensive out-of-the-box accessibility support, which is a critical advantage [[54](https://juejin.cn/post/7633633244707045410)]. When implementing these charts, the most important consideration is the use of color. Qualitative research often involves collaborative work, and ensuring that visualizations are interpretable by individuals with color vision deficiencies is a professional obligation. The provided sources point directly to the **Okabe-Ito palette**, a color scheme specifically designed by researchers to be perceptually uniform and robust under various forms of color blindness [[15](https://stackoverflow.com/questions/57153428/r-plot-color-combinations-that-are-colorblind-accessible), [105](https://arxiv.org/pdf/2303.04918)]. This palette should be the default for all charts generated by the application. Furthermore, reliance on color alone to encode data must be avoided. Visual encoding should be redundant, using patterns, textures, or labels in addition to color. The contrast ratio between text and background must also meet WCAG recommendations to ensure readability for all users [[35](https://accessiblepalette.com/), [36](https://www.w3.org/WAI/WCAG21/Understanding/contrast-minimum.html)].

Implementing accessibility goes beyond just choosing a good color palette. Interactive elements, especially tooltips, must be made keyboard-navigable and compatible with screen readers. A tooltip that appears on hover must also be triggered by keyboard focus (e.g., `focus` event) and should be programmatically associated with the element it describes using ARIA roles and attributes, such as `aria-describedby` [[65](https://stackoverflow.com/questions/45364249/tooltip-accessibility), [106](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/tooltip_role)]. React libraries like React Aria provide foundational support for building widgets that adhere to WAI-ARIA guidelines, simplifying the process of creating an inclusive user interface [[33](https://react-aria.adobe.com/quality)]. Even if a third-party charting library does not handle all aspects of accessibility automatically, developers can use ARIA attributes to enhance its accessibility [[31](https://help.tableau.com/current/tableau/en-us/accessibility_updates.htm)].

For the timeline view, which shows when codes were applied, the choice of library becomes more nuanced. This could represent time by document import date or by the sequential order of documents within the corpus . Libraries like ECharts, mentioned in the context of integration with Electron-React boilerplates, or custom implementations using D3.js, would be capable of rendering such a chart [[93](https://blog.csdn.net/gitblog_00117/article/details/151985091)]. The timeline view's modular approach can also render gaps in work more explicit, helping researchers pinpoint periods of intense or sparse coding activity [[127](https://dl.acm.org/doi/10.1145/3723498.3723821)].

However, a custom implementation using **Visx** or **D3.js** provides the greatest long-term flexibility and ensures pixel-perfect control over both the visual presentation and the underlying accessibility features. The decision should weigh the immediate development speed of a library like Nivo against the future-proofing benefits of a more customized, D3-based solution.

## Dual-Path Rendering Strategy for Network Diagrams

The implementation of force-directed network diagrams presents a significant technical challenge related to performance and interactivity, particularly given the potential scale of the data: up to 500 code nodes and 5,000 edges . The user's insight correctly identifies that a single rendering technology is insufficient to meet all needs. A rigid adherence to SVG will lead to unacceptably slow performance for large graphs, while a pure WebGL approach may compromise the precision and interactivity desired for smaller, exploratory views. Therefore, the optimal strategy is a **dual-path rendering architecture** that intelligently selects the rendering backend based on the size of the graph being displayed.

The first path utilizes **SVG-based rendering** for smaller, filtered subgraphs. When a researcher is exploring relationships between a handful of key codes, the graph will contain fewer than 100 nodes . For this scenario, SVG is the superior choice. SVG renders crisp, scalable vector graphics that look sharp at any zoom level, and it naturally integrates with the DOM, making interactivity straightforward. Libraries like **D3-force** (part of the D3.js suite) are purpose-built for creating SVG-based force simulations [[88](https://d3js.org/d3-force)]. They excel at handling hundreds of nodes and thousands of edges with smooth drag-and-drop interaction, zoom, and pan. The ability to easily add text labels and tooltips to SVG elements is a significant advantage for annotating a small, focused network. The user's expectation that a 50-node graph should be perfectly smooth is easily met by any modern SVG rendering library . This path supports the common analytical workflow of starting with a few hypotheses and exploring the connections between them in detail.

The second path employs **WebGL-accelerated rendering** for the full-corpus view. When a researcher needs to visualize the entire network of 500 codes, the sheer number of nodes and edges makes SVG-based manipulation impractical, as it would require updating thousands of individual DOM elements, leading to severe lag and a jarring user experience [[85](https://stackoverflow.com/questions/7730806/d3-force-layout-visualization-dead-slow-when-using-a-large-dataset)]. WebGL offloads the rendering workload from the CPU to the GPU, enabling the smooth display of tens of thousands of particles or shapes simultaneously [[67](https://pmc.ncbi.nlm.nih.gov/articles/PMC12061801/)]. Libraries like **`react-force-graph`** are specifically designed for this purpose, combining a D3 physics engine with a WebGL renderer to produce highly performant force-directed layouts [[95](https://www.mdpi.com/2071-1050/15/19)]. While WebGL-based graphs may have a slightly different feel—for example, dragging a node might feel less "sticky" than in an SVG simulation—the payoff is the ability to interact with a massive network that would otherwise be completely unmanageable. The initial layout calculation might take longer, but once the graph stabilizes, subsequent interactions like panning and zooming remain fluid [[67](https://pmc.ncbi.nlm.nih.gov/articles/PMC12061801/)]. This mode caters to the exploratory, discovery-oriented phase of analysis, where the goal is to get a holistic sense of the overall structure and identify major clusters or hubs.

This dual-path strategy requires a thoughtful implementation. The application should default to the SVG view, perhaps showing only the top 50-100 codes by frequency to keep the graph readable . The user should have a clear control, such as a toggle button or a dropdown menu, to switch to the WebGL rendering mode for the full graph. This architecture elegantly solves the performance problem by matching the right tool to the right task. It provides the fine-grained interactivity needed for hypothesis testing on small graphs and the scalability required for broad exploration of large ones. The following table compares the key libraries for this dual-path approach:

| Library | Primary Rendering | Best Use Case | Bundle Size Estimate | GitHub Repository |
| :--- | :--- | :--- | :--- | :--- |
| **D3-force** (via D3.js) | SVG | Small, filtered subgraphs (<100 nodes). Provides rich interactivity and text rendering [[88](https://d3js.org/d3-force)]. | Part of D3.js (~500 KB minified) [[95](https://www.mdpi.com/2071-1050/15/19)] | https://github.com/d3/d3-force |
| **`react-force-graph`** | WebGL | Large, full-corpus graphs (>500 nodes). Prioritizes rendering performance over DOM interaction [[95](https://www.mdpi.com/2071-1050/15/19)]. | ~500 KB (dependencies included) | https://github.com/vasturiano/react-force-graph |
| **Cytoscape.js** | SVG/Canvas | Versatile; can handle large graphs via its Canvas extension. Widely used in bioinformatics [[118](https://pubs.acs.org/doi/10.1021/acs.jproteome.2c00651)]. | ~500 KB minified | https://github.com/cytoscape/cytoscape.js |
| **Sigma.js** | Canvas | Optimized for rendering very large graphs with high frame rates. Designed for performance [[87](https://www.sigmajs.org/)]. | ~200 KB minified | https://github.com/sigmajs/sigma.js |

In summary, the recommended implementation is not to choose a single library but to architect the application to conditionally render the network diagram using either D3-force/SVG for small views or `react-force-graph`/WebGL for large views. This hybrid approach provides the best of both worlds, offering a responsive and interactive experience across the full spectrum of analytical tasks.

## Dashboard Architecture and Data Freshness Management

The architecture of the analytics dashboard has profound implications for the overall performance, usability, and resource consumption of the desktop application. The choice of how to present analytical views—whether as inline panels, a flexible mosaic layout, or as separate windows—is a critical design decision that must align with the application's performance constraints and the established UX pattern for data freshness. Given that the application is built with Electron, which bundles a full Chromium instance, managing memory and CPU usage is paramount [[23](https://electronjs.org/), [70](https://electronjs.org/docs/latest/tutorial/performance)].

Three primary architectural models were considered: inline panels within the main window, a detachable panel system, and a separate `BrowserWindow` for each analytics view.

Inline panels are the simplest to implement but carry significant risks. Rendering large, complex visualizations like heatmaps or WebGL graphs within the main application window means their resources are tied to the lifetime of the main process. Even if a panel is collapsed or hidden, the heavy JavaScript libraries and their associated DOM/CSSOM trees and canvas/WebGL contexts may remain in memory, consuming resources unnecessarily. This approach tightly couples the performance of the editor with the performance of the visualizations, which is undesirable.

A detachable panel system, using a library like Golden Layout or React Mosaic, offers greater user flexibility, allowing them to arrange the workspace according to their preference . However, this complexity comes at a cost. Managing the state of multiple, independent visualization components can become intricate. More importantly, it doesn't fundamentally solve the resource management problem. Each visualization component, when mounted, will still consume its own memory and CPU, and there is no easy mechanism to completely unload and garbage collect a visualization when it is moved out of the main window.

The most robust and performant architecture is to implement the analytics dashboard as a **separate Electron `BrowserWindow`** . This approach provides complete isolation of concerns. When a user requests an analytics view (e.g., "Show Co-occurrence Matrix"), the application creates a new `BrowserWindow`. This new window runs in its own Chromium process, separate from the main editor. This separation yields several key benefits:
*   **Memory Management:** If the user closes the analytics window, the entire rendering context—including the heavy D3.js or WebGL libraries—is destroyed, and its memory is returned to the system. This prevents the accumulation of unused resources.
*   **Performance Isolation:** Heavy computation and rendering for charts occur in a background process and do not block the main UI thread of the editor, ensuring a fluid coding experience.
*   **Data Freshness UX:** The "dirty state" paradigm maps perfectly to this architecture. The detached window can clearly display a banner indicating that its results are outdated. Clicking this banner can trigger a refresh, which operates asynchronously in the background, leaving the main editor untouched. This cleanly separates the analytical workflow from the coding workflow .

Regarding data freshness, the user's directive is unequivocal: expensive analytical views must **not** update in real-time. Live recomputation on every coding change is rejected as both computationally expensive and unnecessary, as coding and analysis are temporally distinct activities for researchers . The correct pattern is a "dirty state" indicator. When new annotations are saved, the analytics dashboard (whether in a separate window or panel) detects this and updates its display to show a prominent message like "Results are outdated — click to refresh." This provides honest feedback about data freshness without initiating any background processing. The actual computation is deferred until the user explicitly signals their readiness to analyze by clicking the refresh button. This approach guarantees that the primary coding interface remains snappy and responsive.

Conversely, a narrow set of cheap statistics—such as the total number of annotations, the count of unique codes used, or the number of documents with at least one annotation—can and should update in real-time in a status bar . These metrics are derived from simple, low-cost queries (O(1) or O(n)) and serve as valuable progress indicators for the coder. Their reactivity does not impose a performance penalty on the core workflow. A hard architectural line must be drawn between these live-updating counters and the expensive, on-demand analytical views to prevent contamination of the design and maintain optimal application performance.

## Feature Prioritization Informed by QDA Tool Usability

The final dimension of the implementation plan involves prioritizing the development of the proposed visualizations based on empirical evidence of their utility to qualitative researchers. While the planned features—co-occurrence matrix, code frequency charts, and network diagrams—are theoretically powerful, their value is ultimately determined by how researchers actually use them in practice. A review of academic literature and expert commentary on leading commercial QDA software provides a framework for making informed decisions about which features to implement first in the v2 release.

Leading QDA tools like NVivo, ATLAS.ti, and MAXQDA all incorporate a variety of visualization features, but their prominence and perceived utility differ. The provided source materials, which include numerous articles, reviews, and software comparisons, highlight that these tools are widely used for transforming qualitative data into findings and analyzing coded data [[9](https://www.linkedin.com/pulse/mixed-methods-data-analysis-software-research-borg-ellul-dprof-3qgzf), [37](https://eliteresearch.com/the-future-of-qualitative-data-analysis-a-comprehensive-review-of-atlas-ti-dedoose-nvivo-and-maxqda/), [41](https://www.academia.edu/39694110/Qualitative_Analysis_Software_ATLAS_ti_Ethnograph_MAXQDA_NVivo_)]. For instance, ATLAS.ti prominently features a "Co-occurrence Explorer" tool, suggesting that this type of analysis is considered a core function for its user base [[26](https://atlasti.com/research-hub/co-occurrence-analysis-with-atlas-ti), [120](https://doc.atlasti.com/ManualWin.v9/ATLAS.ti_ManualWin.v9.pdf)]. Similarly, NVivo supports mixed methods research by enabling the analysis of qualitative data alongside quantitative outputs, implying a deep integration of different analytical perspectives [[44](https://atlasti.com/guides/the-guide-to-mixed-methods-research/software-tools-for-mixed-methods-data-analysis)]. MAXQDA is noted for its excellent functions relating to viewing data after it has been coded, indicating a focus on data inspection and interpretation tools [[2](https://www.researchgate.net/post/How_does_Atlasti_and_MAXQDA_perfom_in_qualitative_data_analysis)].

However, a critical gap exists in the provided context: none of the snippets contain direct citations from peer-reviewed usability studies or forum discussions (e.g., from CAQDAS.net) that explicitly rank the popularity or utility of specific visualizations like the co-occurrence matrix versus the network diagram. This information is essential for objective prioritization. Without it, feature implementation is based on theoretical value rather than demonstrated demand. Therefore, the development team must conduct a secondary literature search or informal survey of existing QDA user communities to gather this missing evidence before committing to a full feature set.

Despite this data gap, we can infer priorities based on general principles of qualitative methodology and the stated goals of QDA software. Thematic analysis, a widely used method, involves identifying, analyzing, and reporting patterns within data [[7](https://journals.sagepub.com/doi/10.1177/16094069251384401), [110](https://www.researchgate.net/publication/390799104_Using_thematic_analysis_in_qualitative_research)]. The foundational tools for this process are likely to be most universally valuable.

Based on this reasoning, the following prioritization is recommended for the v2+ roadmap:

1.  **High Priority: Code Frequency Charts.** This category includes simple bar charts and stacked bar charts by document. These are essential descriptive statistics that form the bedrock of almost any qualitative analysis. Researchers need to know which codes appear most frequently and how they are distributed across their data sources. This feature is fundamental and has a low implementation risk, making it an ideal starting point.

2.  **Medium-High Priority: Co-occurrence Matrix.** This visualization is a cornerstone of pattern detection, moving beyond simple frequency to reveal the relationships and associations between concepts (codes). Its inclusion in the core feature sets of major commercial tools like ATLAS.ti and its alignment with the industry-standard definition from NVivo suggest it is a highly valued and expected feature for serious analytical work [[45](https://pmc.ncbi.nlm.nih.gov/articles/PMC12796141/), [120](https://doc.atlasti.com/ManualWin.v9/ATLAS.ti_ManualWin.v9.pdf)]. Implementing this after the basic frequency charts will significantly enhance the application's analytical depth.

3.  **Medium Priority: Timeline View.** This chart type, showing the temporal application of codes, is a valuable but more niche feature. It is particularly useful for longitudinal studies or for tracking a researcher's own coding process over time [[127](https://dl.acm.org/doi/10.1145/3723498.3723821)]. While powerful, it may be less universally applicable than the co-occurrence matrix, justifying its placement as a later-stage feature.

4.  **Low Priority (v2.1+): Network Diagrams.** Force-directed network diagrams are powerful exploratory tools, but they are also more complex for novice users to interpret. They represent a more advanced form of analysis. Commercial software like ATLAS.ti includes them, but they are often positioned as "power-user" features [[48](https://atlasti.com/)]. Given the project's scope and the need to establish a stable foundation, implementing the network diagram as a v2.1 feature, potentially enabled by an advanced setting, is a prudent strategy. This allows the core analytical capabilities to be solidified first, with the more experimental and complex visualization added later based on user feedback and demonstrated need.

By adopting this phased approach, the development team can focus on delivering the most impactful and broadly useful features first, establishing a solid foundation for the analytics layer. This user-centric, evidence-informed prioritization ensures that development resources are allocated to features that provide the greatest value to the target audience of qualitative researchers.
